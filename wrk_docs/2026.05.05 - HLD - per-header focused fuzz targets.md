### HLD — Per-header focused fuzz targets

**Date:** 2026-05-05
**Author:** Claude (with Arthur)
**Status:** Draft for review
**Related:** `2026.05.04 - HLD - SIP parser round-trip oracle.md`,
`2026.05.05 - HLD - SDP parser round-trip oracle.md`,
`fuzz/fuzz_targets/sip_message.rs`, `fuzz/fuzz_targets/sip_via.rs`,
`fuzz/fuzz_targets/sip_uri.rs`

## Context

We fuzz SIP across two separate fuzz crates today:

- `fuzz/` (top-level, multi-target suite) — has `sip_message`
  (whole-message crash), `sip_uri` (`SipUri::parse`), `sip_via`
  (legacy `headers::Via::parse`), `sip_digest`, plus the broader
  RTP/RTCP/SDP/codec targets. Corpora here are gitignored
  (`fuzz/.gitignore` lists `corpus`) and currently absent on disk.
- `crates/rsiprtp/fuzz/` (M11 supervisor's home) — only three
  targets: `sip_message_parse`, `sip_message_parse_diff` (the
  differential oracle against `rsip 0.4`), and
  `sip_message_roundtrip`. Has `rsip 0.4` as a regular dep to
  drive the differential. The active 8h supervisor
  (`fuzz_supervisor.ps1`) only runs `sip_message_parse`.

Both kinds of whole-message target spend most of their mutation
budget on the request line, the framing layer, and `Content-Length`
accounting. Header values past the start-line are gated behind a
successful framing parse; the fuzzer reaches deep header-grammar
paths only by chance. That's the early-reject hiding effect another
agent flagged.

The actual canonical typed-header parsers live in
`crate::sip::parser::typed::{Via, Contact, From, To, CSeq}` plus
the shared `crate::sip::parser::name_addr::NameAddr` they delegate
to. These are what `SipMessage::parse` calls into post-framing — the
*real* per-header grammars. Today only one of them (`Via`,
indirectly via the legacy `headers::Via` wrapper) gets focused
fuzzing.

Per-primitive targets land in `fuzz/` (the multi-target suite),
not `crates/rsiprtp/fuzz/` (M11-specific). They sit alongside the
existing `sip_via` / `sip_uri` / `sip_digest` targets.

## Goals

- Add focused libfuzzer targets that exercise each header parser
  directly with no framing surface ahead of them, so each grammar
  saturates on its own coverage map.
- Match the existing `sip_uri` / `sip_via` shape (pure crash/UB
  targets, ~7 LOC each) — no oracle, no skip mechanism, no feature
  flags.
- Land in `fuzz/` alongside the existing per-primitive targets.
  Don't touch `crates/rsiprtp/fuzz/` (M11 supervisor's home).

## Non-goals

- Round-trip or differential oracles per typed header. Worth doing,
  separate change. The crash target is the cheap step that pays for
  itself first.
- Fuzz coverage of every header type. The four grammars chosen
  (`Via`, `Contact`, `NameAddr`, `CSeq`) are the load-bearing ones
  parsed by `SipMessage::parse`. `Authorization`, `WWW-Authenticate`,
  `Route`/`Record-Route`, `Supported`, `Require`, session-timer
  headers, etc. can follow the same template later if there's signal
  that they need it.
- Fuzz the legacy `crate::sip::headers::Contact::parse`. It's a
  parallel public-surface parser (same situation as legacy `Via`),
  not currently fuzzed. Out of scope here; flagged as a known gap.
- Standalone fuzz target for `parser::method::Method::from_str`.
  Exercised transitively via `sip_cseq` and the whole-message
  targets; its grammar (case-insensitive method tokens + extension
  methods) is small enough that a dedicated target is overkill.
- Fix the legacy `crate::sip::headers::Via` parser duplication. The
  rewrite is on the M-series roadmap; this change does not depend
  on it.
- Replace or merge the existing `sip_via` target. It covers a
  distinct (legacy, publicly callable) parser surface — retiring
  it is a separate decision.

## What's already in place

- `parser::typed::Via::parse(&str) -> Result<Via, SipError>` in
  `crates/rsiprtp/src/sip/parser/typed/via.rs:62`.
- `parser::typed::Contact::parse(&str) -> Result<Contact, SipError>`
  in `crates/rsiprtp/src/sip/parser/typed/contact.rs:113`. Two
  branches: wildcard (`*[;param]`) and `NameAddr`-delegating.
- `parser::name_addr::NameAddr::parse(&str) -> Result<NameAddr, SipError>`
  in `crates/rsiprtp/src/sip/parser/name_addr.rs:65`. The shared
  display-name + addr-spec + header-params grammar.
- `parser::typed::From::parse` and `parser::typed::To::parse` —
  *both* are 1:1 wrappers around `NameAddr::parse` that just
  re-shape the result into a typed struct (see `from.rs:39`,
  `to.rs:30`). They contain zero distinct parser logic. Fuzzing
  `NameAddr::parse` directly covers their parser surface entirely.
- `parser::typed::CSeq::parse(&str) -> Result<CSeq, SipError>` in
  `crates/rsiprtp/src/sip/parser/typed/cseq.rs:36`. Internally calls
  `parser::method::Method::from_str`, so that grammar is hit
  transitively.
- `crate::sip::parser` is `pub mod parser` with `#[doc(hidden)]`
  (see `crates/rsiprtp/src/sip/mod.rs:14`); inside it,
  `pub mod typed` and `pub mod name_addr` (see
  `parser/mod.rs:24,26`). So the fuzz crate can reach all four
  parsers directly via `rsiprtp::sip::parser::*` without any new
  re-exports. The same backdoor is already used by
  `tests/parser_diff.rs`.
- `fuzz/Cargo.toml` already wires `rsiprtp = { path = "../crates/rsiprtp" }`.

So the wiring exists. What's missing is four small target files and
their `[[bin]]` stanzas.

## Design

### Targets

| Bin name        | Wire target                                  | Source file (new)                           |
| --------------- | -------------------------------------------- | ------------------------------------------- |
| `sip_via_typed` | `parser::typed::Via::parse(&str)`            | `fuzz/fuzz_targets/sip_via_typed.rs`        |
| `sip_contact`   | `parser::typed::Contact::parse(&str)`        | `fuzz/fuzz_targets/sip_contact.rs`          |
| `sip_name_addr` | `parser::name_addr::NameAddr::parse(&str)`   | `fuzz/fuzz_targets/sip_name_addr.rs`        |
| `sip_cseq`      | `parser::typed::CSeq::parse(&str)`           | `fuzz/fuzz_targets/sip_cseq.rs`             |

Naming follows the existing `sip_<noun>` convention (`sip_uri`,
`sip_via`, `sip_digest`). The agent's suggestion used `parse_<noun>`;
matching the in-tree style is the cleaner call.

Each target shape mirrors `fuzz_targets/sip_via.rs`:

```rust
#![no_main]
use libfuzzer_sys::fuzz_target;

fuzz_target!(|data: &[u8]| {
    if let Ok(s) = std::str::from_utf8(data) {
        let _ = rsiprtp::sip::parser::typed::Contact::parse(s);
    }
});
```

All four targets follow this exact shape. No discriminator bytes,
no per-target oddities.

### Why `sip_name_addr` instead of `sip_from_to`

The agent's suggestion was `parse_from_to`. Looking at the actual
parsers, `From::parse` and `To::parse` are 1:1 wrappers around
`NameAddr::parse` — neither contains a single line of distinct
parser logic past the delegation call (see `from.rs:39-46`,
`to.rs:30-37`). A `sip_from_to` target with a discriminator byte
would spend half its mutation budget on identical code with
identical coverage signal.

Targeting `NameAddr::parse` directly:
- Same parser surface as From/To (and the non-wildcard branch of
  Contact, by extension).
- One coherent corpus instead of two parallel ones discriminated
  on a coin-flip byte.
- No ambiguity about which wrapper a corpus entry was found
  through.

If `From::parse` or `To::parse` ever grow header-specific parser
logic past the NameAddr delegation, we add wrapper-specific
targets at that point.

### Why no new `sip_uri` target

`SipUri::parse` is already covered by the existing `sip_uri`
target. `NameAddr::parse` (and therefore the typed
`From`/`To`/`Contact-Addr` paths) calls into it — so the URI
surface is hit transitively from `sip_name_addr` and `sip_contact`
*and* directly from `sip_uri`. No duplicate target needed.

### Why keep `sip_via` and add `sip_via_typed` rather than retarget

`sip_via` calls `crate::sip::headers::Via::parse` — the legacy
public-surface parser still reachable via the
`crate::sip::Via` re-export. The typed `parser::typed::Via::parse`
is a distinct re-implementation: stricter `sent-protocol` ABNF,
ordered `(key, Option<String>)` params, IPv6-bracket-preserving
`sent_by`. Two parsers, two grammars; both have public-surface
exposure today. Adding `sip_via_typed` alongside `sip_via` covers
both. If/when the legacy `headers::Via` is deleted, the old
`sip_via` target retires with it.

(Earlier draft argued for "preserving saturated corpus" — that's
moot: `fuzz/.gitignore` excludes `corpus/` and the directory is
empty on disk. The two-distinct-parsers argument stands on its
own.)

### Corpus seeding

libfuzzer will start from empty corpora and bootstrap. We can give
each target a small head start without much effort by scraping the
corresponding header values out of the existing fixtures in
`crates/rsiprtp/tests/fixtures/sip/` and `crates/rsiprtp/src/sip/tests/`.
Optional — listed as a follow-up step in the implementation plan,
not a blocker. The whole-message target seeded itself fine from
empty.

### Skip / pin mechanism

**None.** Same call as the round-trip oracles: these are
pure crash/UB targets. A panic, OOM, or hang is a real bug. If a
known issue surfaces and we choose not to fix it immediately, pin
it via the same approach used for M11 fuzz findings #16/#17 (commit
`407de58`).

## Trade-offs and acknowledged risks

**Risk:** four new targets compete with the existing per-primitive
SIP/SDP fuzz targets (`sip_message`, `sip_via`, `sip_uri`,
`sip_digest`, `sdp_session`) for CPU during overnight soaks of the
top-level `fuzz/` suite. Adding four splits the budget further.
Worth calling out at the next campaign-planning step; the targets
are still cheap to *build* and a future supervisor can drop the
duplicate `sip_via` once the legacy parser retires.

**Trade-off rejected:** add a typed-header round-trip oracle in
the same change. Each of the typed forms has a `Display` impl,
so the oracle pattern from M11 / SDP-RT is portable. Worth doing,
*not* in this change — bundling expands the review surface for no
schedule benefit, and the crash target is the simpler win to land
first.

**Trade-off rejected:** retarget `sip_via.rs` to the typed parser
in this change. Covered in "Why keep `sip_via` and add
`sip_via_typed`" above — the legacy parser is still publicly
callable and is its own grammar; conflating them under one target
name loses signal.

**Trade-off rejected:** also add `sip_contact_legacy` for
`crate::sip::headers::Contact::parse`. It's a parallel
public-surface parser with the same stale-grammar property as
legacy Via, currently unfuzzed. Skipping it here to keep the
change focused on the typed parsers; flag as a known gap.

**Trade-off rejected:** add a `sip_method` target for
`parser::method::Method::from_str`. The grammar is
case-insensitive method tokens plus extension methods — small
enough that `sip_cseq` (which calls into it) and `sip_message`
provide adequate coverage. Note for future if signal warrants.

**Trade-off rejected:** add fuzz targets for `Authorization`,
`WWW-Authenticate`, `Route`/`Record-Route`, `Supported`, `Require`,
session-timer headers. None of those have typed parsers in
`parser::typed/` yet (some live in `crate::sip::headers`, some are
free functions on `SipMessage`). Adding fuzz coverage there is a
follow-up tied to the typed-parser rewrite, not this change.

**Trade-off rejected:** cross-target deduplication (e.g. shared
`fn parse_str_target<F>(...)`). Each target is 7 lines; the
abstraction would be larger than the duplication.

## Out of scope

- Round-trip oracles for typed headers.
- Per-header differential targets against `rsip 0.4`.
- Wrapper-level targets for `From::parse` / `To::parse`
  (covered by `sip_name_addr`).
- Fuzzing legacy `crate::sip::headers::Contact::parse`.
- Standalone `Method::from_str` target.
- Retiring or retargeting `sip_via`.
- Anything in `crates/rsiprtp/fuzz/` (M11 supervisor's home).
- Supervisor / CI wiring for the new targets — campaign planning
  concern, separate change.

## Prerequisites

Same as the existing fuzz targets — these are cargo-fuzz
prerequisites, not new requirements introduced by this change. Per
`crates/rsiprtp/fuzz/M11_LAUNCH.md`:

- **Nightly Rust toolchain.** `cargo-fuzz` only works on nightly
  (`rustup toolchain install nightly`).
- **`cargo-fuzz`** ≥ 0.13 (`cargo install cargo-fuzz`).
- **Windows MSVC only:** the fuzz binaries dynamically link
  `clang_rt.asan_dynamic-x86_64.dll`. Add the MSVC clang_rt
  directory to `PATH` before launch (see `M11_LAUNCH.md` for the
  exact path on Arthur's box).

## Implementation plan

1. Write four target files in `fuzz/fuzz_targets/` (each ~7 LOC):
   `sip_via_typed.rs`, `sip_contact.rs`, `sip_name_addr.rs`,
   `sip_cseq.rs`.
2. Append four `[[bin]]` stanzas to `fuzz/Cargo.toml`.
3. From `d:\language\rsiprtp\fuzz`: `cargo +nightly fuzz build
   <target>` for each of `sip_via_typed`, `sip_contact`,
   `sip_name_addr`, `sip_cseq` — confirm all four compile.
   (`cargo fuzz build` does not accept multiple bin names per
   invocation.)
4. From the same directory: `cargo +nightly fuzz run <target> --
   -max_total_time=120` for each target (smoke each for 2 minutes
   one-by-one to catch trivial breakage). On Windows, ensure the
   asan DLL directory is on `PATH` first.
5. Commit. Hand to the next overnight rotation of the top-level
   `fuzz/` suite (separate planning concern).

Total estimated diff: ~30 LOC across 4 new files + ~28 lines added
to `fuzz/Cargo.toml`. No changes to existing code.

## Open questions

- After first 8h soak, is `sip_name_addr`'s coverage strictly
  greater than the NameAddr-reachable subset of `sip_contact`'s?
  If they're indistinguishable, consider folding `sip_name_addr`
  into `sip_contact` (drop one target) — but only with data, not
  on speculation.
- Does `sip_via_typed` find anything `sip_via` misses? If a
  fortnight of soak shows zero divergent crashes, that's signal
  the legacy parser is no more brittle than the typed one — useful
  for the eventual deletion decision.
