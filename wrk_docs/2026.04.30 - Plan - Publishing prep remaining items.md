### 9. Test isolation bug fix

Root cause: `force_*_error` tests use `static AtomicU64` flags storing
thread IDs to inject errors. Parallel tests on the same hashed thread
cause cross-talk.

**Approach options (TBD during implementation):**
- A. Replace shared static flags with per-test scoped state (e.g.,
  `tokio::task_local!` or function-arg injection).
- B. Use a single `Mutex<Option<TestId>>` with explicit lock-and-set in
  each test, ensuring serial execution of just these tests via
  `#[serial_test::serial]`.
- C. Build a small `ErrorInjector` value that's passed into the unit
  under test, eliminating global state entirely.

C is the right end state but biggest refactor. A is probably the path of
least resistance. I'll decide once I read the call sites.

Affects: `rsiprtp-transport::{tcp,tls}`, `rsiprtp-ice::turn` (per
journal). Maybe a couple more.

**Validation:** `cargo test --workspace --exclude gabby` (no
`--test-threads=1`) passes consistently — repeat 5×.

## Risks

- **Item 7 scope blow-out:** could be days of doc-writing alone. Mitigation:
  enforce on facade only first; defer internal crates if too large.
- **Item 9 design choice:** wrong approach could bloat tests or hide bugs.
  Mitigation: pick smallest-scope option (A) first; revisit only if
  problems persist.
- **Item 10 unresolvable:** very likely we never pin down the exact cause.
  Mitigation: ship branch protection + monitoring instead of forensics.
- **MSRV regression mid-plan:** an item we add (cargo-deny? fuzz target?)
  could pull in deps that need newer Rust. Mitigation: re-run the MSRV
  build at end.
