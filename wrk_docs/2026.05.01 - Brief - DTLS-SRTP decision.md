# 2026.05.01 - Brief - DTLS-SRTP decision

## One-line goal

**Decide:** delete the DTLS-SRTP stub or commit to a real DTLS handshake
implementation. The current half-state is the worst option of the three.

## Why this is a decision, not a task

`src/srtp/dtls.rs` (990 lines) advertises DTLS-SRTP support but the
file's own header (lines 6–10) reads:

> **Note:** This module provides the foundational types for DTLS-SRTP but
> does NOT include a full DTLS handshake implementation. The actual DTLS
> transport layer would require integration with a TLS library (e.g.,
> rustls, openssl, or ring).

What exists:

- `Fingerprint` — SHA-256/SHA-1 cert fingerprint parse/format for SDP.
- `DtlsRole` — `active` / `passive` / `actpass` from `a=setup`.
- `SrtpProfile` — five RFC 5764 protection profiles enumerated.
- `DtlsSrtpKeys` — struct to hold exported keying material with role-aware
  `local_keys()` / `remote_keys()` accessors.
- `extract_srtp_keys` — derive SRTP master key/salt from a hypothetical
  DTLS-exported keying buffer.
- `parse_use_srtp_extension` / `build_use_srtp_extension` — DTLS-SRTP
  extension wire format (RFC 5764 §4.1.1).

What does **not** exist:

- The DTLS handshake state machine (ClientHello, HelloVerifyRequest with
  cookie, ServerHello, Certificate, ServerKeyExchange, ServerHelloDone,
  ClientKeyExchange, ChangeCipherSpec, Finished — both sides).
- DTLS record layer over UDP (epoch + sequence number, MAC, AEAD).
- Retransmission and fragmentation per RFC 6347 §4.2.
- Cookie exchange for DoS protection.
- Certificate generation (we cannot self-sign without it).
- Keying material export per RFC 5705 driven by the actual handshake.

That is, in a phrase: *all the easy parts and none of the hard ones.*
The risk is shipping `rsiprtp 1.0` with code that **looks** like
WebRTC interop but isn't. Better to make the boundary explicit.

## The three options

### Option A — Remove (recommended)

Delete `src/srtp/dtls.rs` and the `srtp::dtls` re-exports. Update the
README to say "SDES-SRTP only; no DTLS-SRTP, no WebRTC interop." If
DTLS-SRTP is ever needed, do it as a real feature with a milestone, not
a half-implementation.

**Cost:** ~1 day. Code deletion + module re-export cleanup + README + a
deprecation note in CHANGELOG.

**Wins:**
- Honest scope. README and code agree.
- Smaller public API surface to maintain across 0.x → 1.0.
- Fewer sentences hedging in user-facing copy.

**Loses:**
- The fingerprint parser and `SrtpProfile` enum are useful for users
  doing their own DTLS — those users now have to roll their own.

### Option B — Adopt an external DTLS crate

Lean on `rustls` (which now has experimental DTLS support via the
`rustls-tokio-stream` ecosystem) or the in-tree DTLS implementations in
the `webrtc-rs` project.

**Cost:** 2–4 weeks. Integration is real work even when the handshake
itself is delegated:
- Building a DTLS-over-UDP transport on top of our existing socket layer.
- Bridging DTLS keying material to our `SrtpContext`.
- Handling the role coupling between `a=setup` (SDP) and the DTLS
  handshake (who sends ClientHello).
- ICE coupling: DTLS handshake starts after ICE succeeds, on the
  selected candidate pair. So this **blocks behind ICE wiring**.

**Wins:**
- WebRTC interop becomes possible. Browser softphones can talk to us.
- We get a real feature for the marketing claim.

**Loses:**
- Significant ongoing maintenance: `rustls` DTLS is not stable; webrtc-rs
  is a moving target.
- Adds a heavy dependency to a stack that is currently lean.
- Pulls in cert generation choices (rcgen) and probably a self-signed
  cert lifecycle.

### Option C — Keep the stub as-is

**Cost:** zero today, growing.

**Wins:** none.

**Loses:**
- Misleads users into thinking DTLS-SRTP works when it doesn't.
- Every README/doc pass has to remember to qualify the claim.
- Anyone writing tests against this module can't actually use it.

This option exists only to be named and rejected.

## Recommendation

**Option A — remove.** Ship 1.0 with an honest "SDES-SRTP only" claim,
which is correct and useful for the entire SIP-trunking / VoIP audience.
Re-add DTLS-SRTP as a deliberate v1.x feature once we have a customer
pulling in that direction, ICE is wired, and we've picked a DTLS crate
to bet on. The decision becomes much easier when there is real demand
and not just a feeling that "WebRTC support would be nice."

If Arthur wants to keep the door open with minimum cost, Option A.5: keep
*only* `Fingerprint` and `SrtpProfile` (genuinely useful primitives) and
move them to `srtp/sdes.rs` or a new `srtp/common.rs`. Delete everything
else from `dtls.rs` along with the module entirely. This is a 1-day job.

## What "remove" actually entails

1. Delete `crates/rsiprtp/src/srtp/dtls.rs`.
2. Remove `pub mod dtls;` (or `pub(crate) mod dtls;`) from `srtp/mod.rs`.
3. Remove any `pub use` re-exports of `Fingerprint`, `DtlsRole`,
   `SrtpProfile`, `DtlsSrtpKeys`, `DtlsState`, `DtlsError`,
   `extract_srtp_keys`, `parse_use_srtp_extension`,
   `build_use_srtp_extension`.
4. Search for downstream uses (`grep -r dtls`):
   - `crates/rsiprtp/src/lib.rs` — top-level doc references DTLS-SRTP.
     Trim to "SRTP via SDES key exchange (RFC 4568)."
   - `README.md` — update the SRTP/DTLS line in Features and the
     bullet at the top of Scope.
5. Update CHANGELOG: a clear line about removed surface — pre-1.0 so
   no semver promise broken, but worth flagging.
6. `cargo build && cargo clippy --all-targets -- -D warnings && cargo test`.

If we keep `Fingerprint` and `SrtpProfile` (Option A.5), instead:
1. Move them to `srtp/profile.rs` (new file).
2. `pub use` them from `srtp/mod.rs`.
3. Update `srtp/sdes.rs` to use the relocated `SrtpProfile` if it
   currently has its own copy (check first).
4. Same delete-everything-else cleanup as above.

## Open questions for Arthur

- **Is there a known user / use case that needs DTLS-SRTP?** If yes,
  Option B becomes viable. If not, Option A is the right call.
- **Do we want to publish 1.0 with WebRTC explicitly out of scope, or
  position this as "SIP-only stack, WebRTC sister-crate later"?** This
  shapes the README copy.
- **CHANGELOG framing.** Pre-1.0, removal is fine, but a clear note
  prevents surprise for anyone who pinned 0.2 expecting DTLS support.

## What this brief is *not*

This is not a plan for implementing DTLS-SRTP. If we go Option B, that
needs its own brief, an HLD, and probably a spike to validate the chosen
DTLS crate before committing. For now: a yes/no on whether the stub
stays.
