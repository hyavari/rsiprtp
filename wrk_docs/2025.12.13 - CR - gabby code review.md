# Gabby Voice AI Agent - Code Review Report

**Date:** 2025-12-13
**Reviewer:** Claude Code
**Crate:** `crates/gabby`
**Status:** Initial Implementation Review

---

## Executive Summary

Gabby is a voice AI SIP agent that enables natural phone conversations using Vosk (STT), Ollama (LLM), and Piper (TTS). The implementation provides a functional proof-of-concept with a well-organized codebase. However, several issues need attention before production use:

- **Critical Issues:** 2 (SIP parsing fragility, audio pipeline timing)
- **High Issues:** 7 (security, protocol compliance, resource management)
- **Medium Issues:** 12 (code quality, edge cases, duplication)
- **Low Issues:** 8 (documentation, minor improvements)

**Overall Assessment:** The codebase demonstrates solid Rust fundamentals and appropriate use of async patterns. The architecture is sound, but the implementation takes shortcuts in SIP protocol handling that could cause interoperability issues. Recommended for demo/testing purposes; requires hardening for production.

### 4. src/server.rs

**Location:** `crates/gabby/src/server.rs`
**Purpose:** SIP server and call management

**Findings:**

| Severity | Issue | Line | Description |
|----------|-------|------|-------------|
| **Critical** | Fragile SIP parsing | 328-337 | `extract_header` uses string manipulation that can fail on multi-line headers, quoted strings, or compact headers. |
| **High** | No authentication | 104-241 | Any caller can connect - no SIP authentication (Digest, etc.). |
| **High** | Protocol non-compliance | 390-414 | `build_response` hardcodes `sip:caller@remote` instead of using actual From header. |
| **High** | Port collision | 171-172 | `rtp_port_counter` wraps at 1000, could collide with active calls. |
| Medium | Header extraction bug | 333 | Uses `name.len() + 1` but comparison was case-insensitive - could slice incorrectly. |
| Medium | SDP parsing | 350-361 | Minimal SDP parsing doesn't validate format, could parse wrong line. |
| Medium | CANCEL response | 295-309 | CANCEL handler doesn't send 200 OK response back to caller. |
| Medium | Memory leak | 201 | `active_calls` HashMap entries only removed on BYE - leaked on CANCEL or crash. |
| Low | Via header | 127-129 | Via branch not validated (should start with "z9hG4bK" per RFC 3261). |

**Critical Issue Detail - SIP Parsing:**

The current parsing:
```rust
fn extract_header(msg: &str, name: &str) -> Option<String> {
    let name_lower = name.to_lowercase();
    for line in msg.lines() {
        let line_lower = line.to_lowercase();
        if line_lower.starts_with(&format!("{}:", name_lower)) {
            let value = line[name.len() + 1..].trim();  // BUG: uses original name length
            return Some(value.to_string());
        }
    }
    None
}
```

Problems:
1. Case-insensitive comparison but case-sensitive slicing
2. Doesn't handle header folding (RFC 3261 allows multi-line headers)
3. Doesn't handle quoted strings with colons
4. Doesn't handle compact header forms beyond simple mapping

**Recommendation:** Use the mdsiprtp-sip crate for proper SIP parsing instead of manual string manipulation.

**Positive Notes:**
- Good async architecture with tokio
- Proper separation of INVITE/BYE/CANCEL handling
- SDP answer construction is functional

### 5. src/call.rs

**Location:** `crates/gabby/src/call.rs`
**Purpose:** Per-call audio pipeline

**Findings:**

| Severity | Issue | Line | Description |
|----------|-------|------|-------------|
| **Critical** | Audio timing mismatch | 188-212 | STT buffer processes in 3200-sample chunks but VAD uses original 160-sample frame - decisions based on wrong audio. |
| **High** | Unbounded buffer | 56-57 | `output_buffer` (VecDeque) and `stt_buffer` (Vec) can grow unbounded if TTS produces faster than playback. |
| **High** | Early greeting | 138 | `send_greeting()` called before ACK received - may cause audio before call established. |
| Medium | No call timeout | 143-170 | No timeout for inactive calls - zombie calls possible. |
| Medium | NAT handling | 289 | Always sends to `remote_rtp_addr` - doesn't handle symmetric RTP or NAT. |
| Medium | Encode algorithm | 321-353 | G.711 mu-law encode implementation differs from standard - potential audio artifacts. |
| Low | Dead code | 34-45 | Multiple `#[allow(dead_code)]` fields suggest incomplete SIP dialog management. |

**Critical Issue Detail - Audio Timing:**

```rust
// In handle_rtp_packet:
let samples: Vec<i16> = payload.iter().map(|&b| decode_ulaw(b)).collect();  // 160 samples @ 8kHz

let resampled = self.input_resampler.process(&samples);  // Now ~320 samples @ 16kHz
self.stt_buffer.extend(resampled);

while self.stt_buffer.len() >= 3200 {  // Process when we have 200ms of audio
    let chunk: Vec<i16> = self.stt_buffer.drain(..3200).collect();
    let partial = self.stt.accept_waveform(&chunk);

    // BUG: VAD processes `samples` (original 160 @ 8kHz) but this is from
    // 10+ frames ago! Should process current audio, not stale buffer.
    let decision = self.vad.process(&samples, partial.as_deref());
```

The VAD decision is made on the most recent 20ms frame but the STT partial is from audio that's 200ms in the past. This temporal mismatch could cause:
- Premature speech-end detection
- Missed speech starts
- Incorrect silence detection

**Recommendation:** Rearchitect to process VAD and STT at the same temporal point, or buffer VAD decisions to match STT timing.

**Positive Notes:**
- Correct G.711 mu-law decode table
- Proper RTP packet construction
- Clean async loop structure

## Recommendations

### Immediate (Pre-Demo)

1. **Fix audio timing mismatch** - Critical for correct operation
   - Align VAD processing with STT temporal window
   - Consider processing VAD on 16kHz resampled audio

2. **Add buffer limits** - Prevent memory exhaustion
   ```rust
   const MAX_OUTPUT_BUFFER_SAMPLES: usize = 8000 * 10; // 10 seconds
   if self.output_buffer.len() > MAX_OUTPUT_BUFFER_SAMPLES {
       self.output_buffer.drain(0..self.output_buffer.len() - MAX_OUTPUT_BUFFER_SAMPLES);
   }
   ```

3. **Limit conversation history**
   ```rust
   const MAX_HISTORY: usize = 20;
   if self.history.len() > MAX_HISTORY {
       self.history.drain(0..2); // Remove oldest exchange
   }
   ```

### Short-Term (Production Readiness)

1. **Use mdsiprtp-sip for SIP parsing** - Proper RFC compliance
2. **Add call timeout** - Clean up zombie calls
3. **Fix CANCEL response** - Send 200 OK per RFC 3261
4. **Add configuration validation** - Fail fast on invalid config
5. **Deduplicate code** - Single calculate_rms, find_sentence_boundary

### Long-Term (Feature Complete)

1. **SIP authentication** - Digest authentication support
2. **NAT traversal** - Symmetric RTP, STUN support
3. **Multiple codec support** - G.722, Opus for better quality
4. **Streaming TTS** - Reduce latency with sentence-by-sentence synthesis
5. **Graceful call teardown** - Send BYE on server shutdown

## Conclusion

Gabby demonstrates a working voice AI concept with solid async Rust architecture. The main concerns are:

1. **Reliability** - SIP parsing fragility could cause interoperability issues
2. **Correctness** - Audio timing mismatch may affect speech recognition quality
3. **Robustness** - Unbounded buffers and no timeouts could cause issues under load

For demonstration purposes with controlled test conditions, the current implementation should work. For production deployment, addressing the critical and high-severity issues is essential.

**Rating:** 6/10 - Functional proof-of-concept requiring hardening for production use.
