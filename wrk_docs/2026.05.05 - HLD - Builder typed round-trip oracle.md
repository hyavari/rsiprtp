# HLD — Builder typed round-trip oracle

**Date:** 2026-05-05
**Author:** Claude (with Arthur)
**Status:** Draft for review
**Related:** `2026.05.04 - HLD - SIP parser round-trip oracle.md`,
`2026.05.05 - HLD - SDP parser round-trip oracle.md`

## Context

M11 landed a bytes-in round-trip oracle:
`bytes → parse → serialize → parse → assert fixed point`. It catches
any non-idempotency in the parse/serialize pair on the parser-layer
types (`crate::sip::parser::{Request, Response}`).

It is structurally blind to two things:

1. **Builder bugs.** `SipRequestBuilder` / `SipResponseBuilder`
   (`crates/rsiprtp/src/sip/message.rs:851, 1318`) format typed
   fields (Via host/port/transport/branch, From URI/tag/display,
   CSeq number+method, RAck, Session-Expires, etc.) into header
   strings. None of that formatting is exercised by the bytes-in
   oracle — its inputs are already wire bytes.
2. **Tier-2 typed-accessor bugs.** `SipRequest::from_tag()`,
   `via_branch()`, `cseq_method()`, etc. round-trip the wire value
   back through `parser::typed::{From, Via, CSeq, …}::parse`. The
   bytes-in oracle never calls those accessors — it only checks
   that the raw header strings round-trip.

A typed round-trip oracle in the *other direction*
(`build typed → serialize → parse → read typed → compare`) closes
both gaps: it asserts the builder formats fields in a way the
parser's typed layer recovers identically.

## Goals

- Catch divergence between the builder's serialize side and the
  parser's typed-accessor side, on a curated set of typed inputs.
- Run as a static-fixture suite. One file, no shared module.
- Land alongside the existing oracles without touching them or the
  builder/parser code.

## Non-goals

- Fuzz the typed-input path. (See *Trade-offs*.)
- Catch bugs that survive both build and read symmetrically (e.g.
  the builder always lowercasing a value the accessor also
  lowercases). That is symmetric loss — same blind spot the
  bytes-in oracle has, same answer: needs grammar conformance, not
  this oracle.
- Test parser-layer struct equality. M11 already does that.
- Change `SipRequestBuilder`, `SipResponseBuilder`, or any
  accessor.

## What's already in place

- `SipRequest::builder()` and `SipResponse::builder()` produce
  typed messages.
- `SipRequest::to_bytes()` / `SipResponse::to_bytes()` go through
  the parser-layer serializer (`crates/rsiprtp/src/sip/message.rs:449,
  686`), so the bytes-in oracle's invariant on the wire form
  applies here too.
- `SipMessage::parse(&[u8])` (`message.rs:154`) parses bytes into
  `SipMessage::Request | Response`.
- Tier-2 typed accessors on the reparsed message:
  `method()`, `uri()`, `call_id()`, `from_tag_and_uri()`,
  `to_tag()`, `to_uri()`, `via_branch()`, `cseq()`, `cseq_method()`,
  `contact_uri()`, `body()`, `content_type()`, `require()`,
  `supported()`, `session_expires()`, `min_se()`, `rack()`,
  `allow()`, `route_headers()`, `via_headers_raw()` — and the
  response equivalents.

So the primitives are all present. What's missing is a test that
runs **builder input → bytes → parse → accessor output** and
asserts the two ends agree.

## Design

### The invariant

For each typed builder input field, the corresponding accessor on
the reparsed message returns a value semantically equal to what
went in. Concretely, for a request:

```rust
let req = SipRequest::builder()
    .method(Method::Invite)
    .uri("sip:bob@example.com")
    .via("alice.example.com", 5060, "UDP", "z9hG4bKnashds7")
    .from("sip:alice@example.com", "1928301774")
    .to("sip:bob@example.com")
    .call_id("a84b4c76e66710@pc33.example.com")
    .cseq(314159)
    .max_forwards(70)
    .build().unwrap();

let r2 = roundtrip_request(req);

assert_eq!(r2.method(), Method::Invite);
assert_eq!(r2.uri().to_string(), "sip:bob@example.com");
assert_eq!(r2.via_branch().unwrap(), "z9hG4bKnashds7");
assert_eq!(r2.from_tag().unwrap(), "1928301774");
assert_eq!(r2.from_uri().unwrap().to_string(), "sip:alice@example.com");
assert_eq!(r2.call_id().unwrap(), "a84b4c76e66710@pc33.example.com");
assert_eq!(r2.cseq().unwrap(), 314159);
assert_eq!(r2.cseq_method().unwrap(), Method::Invite);
```

Equality is at the level of the **typed accessor's return value**,
not the raw header string. That is the strongest assertion the
oracle can make and the one most likely to expose a builder/parser
disagreement.

### Fixture preconditions: inputs must be canonical

The oracle compares accessor output against the fixture's literal
input string. That works only when the input is already in the
canonical form the parse/Display pair will produce. Concretely,
fixtures must obey:

- **URIs in `SipUri::Display` canonical form.** Lowercase scheme
  (`sip:`, not `SIP:`); host case as-given (preserved by parse);
  param order = insertion order. Anything non-canonical fails the
  assertion for normalization, not bug — that's a fixture problem,
  not an oracle finding.
- **Methods that round-trip through `Method` Display↔FromStr.**
  Use the standard set: Invite, Ack, Bye, Cancel, Register,
  Options, Update, Notify, Info, Refer, Subscribe, Publish,
  Message, Prack.
- **CSeq numbers within `0..=2^31-1`.** Builder accepts `u32` but
  the parser-typed CSeq layer may reject overlarge values.
- **Reason phrases free of CTL bytes** (RFC 3261 §25.1, enforced
  by the parser per commit 1c4583a).
- **Tags and branches as RFC token chars.** No spaces, no escapes
  needed.
- **IPv6 hosts pre-bracketed.** The builder's Via emits
  `format!("{host}:{port}")` with no IPv6-awareness, so the caller
  passes `"[::1]"` as host. The parser handles that form
  (verified: `parser::typed::via.rs:329`,
  `crate::sip::uri.rs:697`).

These are not arbitrary restrictions — they're the canonical
forms the production stack itself emits. The oracle's job is not
to test normalization (M11 covers that on the parse side); it is
to test agreement between builder and accessor on already-canonical
inputs.

### What we expect to *not* fixed-point against, and why

The builder fills in defaults when fields are unset. The oracle's
job is not to police those defaults; it's to verify that whatever
the builder *actually emits* round-trips through the parser. So the
test fixtures explicitly set every field they assert on, and the
asserts compare against the **set value**, not against `None`.

Concrete cases to be aware of when writing fixtures (mostly
defaults — listed so the reader doesn't trip over them):

- **Via port defaults to 5060** if `via_port` is unset
  (`message.rs:1146`). Don't assert "input was unset" — assert
  the port the wire form actually carries.
- **Via transport defaults to "UDP"** if unset (`message.rs:1147`).
- **Max-Forwards defaults to 70** if unset (`message.rs:1189`).
- **CSeq method comes from the request method** (`message.rs:1183`).
  CSeq's method on the reparsed request must equal the builder's
  `.method()`.
- **Response reason defaults to canonical phrase** if unset
  (`message.rs:1606`). E.g., `200` becomes `"OK"`.
- **`from_display` is wrapped in literal `"…"` quotes** by the
  serializer (`message.rs:1158`). The reparsed `from_uri()` does
  not return the display name; assert via raw header inspection
  if testing display-name round-trip.
- **Content-Length is rewritten to `body.len()`** by the parser
  serializer (`crate::sip::parser::message::write_headers`,
  documented in M11). The builder also sets it to `body.len()` at
  build time (`message.rs:1301`), so the two agree. The oracle
  doesn't assert against it directly — `body()` carries the
  ground truth.
- **`via_host` may be wrapped** if it contains characters the
  parser interprets as separators. Today it isn't — the builder
  uses `format!("{host}:{port}")` raw — but if a fixture passes a
  hostname with surprising characters and the round-trip changes
  the host, that's a signal worth investigating, not a fixture
  problem.

These are documented expectations, not bugs. Anything that fails
the round-trip outside this list is the kind of bug the oracle is
meant to catch.

### What the oracle is well-positioned to find

- Builder formats a field that the parser's typed layer rejects
  or recovers differently. Example failure mode: a CSeq method
  the builder emits as one form but `parser::typed::CSeq::parse`
  reads as another.
- Builder forgets to emit a field that was set. Example: setting
  `.allow(&[Method::Invite])` and the resulting request has no
  Allow header, or the parser doesn't return it via `.allow()`.
- A new typed field added to the builder without a corresponding
  parser-side accessor. The fixture for that field would fail to
  compile or — worse — silently pass while reading `None`. The
  oracle's structure forces both ends to be wired up.
- `Other`-rendered headers (`Session-Expires`, `Min-SE`, `RAck`,
  `RSeq` — see `message.rs:1240, 1250, 1257, 1571`) where the
  builder uses one name/format and the typed accessor expects a
  different one. The bytes-in oracle would round-trip the raw
  string fine; this oracle catches a mismatch in the typed shape.
- Builder produces bytes the parser rejects outright. The helper's
  first parse step `unwrap`s, so this is a hard failure with a
  pretty-printed diff.

### What the oracle will *not* catch

- **Display name typed read.** The builder emits
  `format!("\"{display}\" <{uri}>;tag={tag}")` for `from_display`
  set, but there is no public `from_display()` accessor. The most
  the oracle can do is byte-match `"\"Alice\""` in `to_bytes()`
  output, which tests the *write* side only. Typed read of
  display name is not exercised.
- **Custom `other_headers`.** `SipRequestBuilder::header(name, value)`
  pushes into `other_headers` for arbitrary names like `Reason:`
  (RFC 3326). There is no typed accessor for these, so the oracle
  has nothing to compare against. Out of scope.
- **Symmetric loss between build and read.** If the builder
  always-lowercases a tag and the accessor always-lowercases on
  read, both ends agree and the oracle is silent. Out of scope;
  needs grammar conformance.

### Reduced isolation power on the response side

`SipResponseBuilder` has **no** `via()` / `from()` / `to()` /
`call_id()` / `cseq()` setters. Via/From/To/Call-ID/CSeq on a
response can only be populated via `.from_request(&req)`
(`message.rs:1351`), which copies header values from a built
request into the response builder's storage and re-emits them.

Consequence: every response fixture that asserts on those headers
transitively exercises the request builder + the copy path + the
response build. A failure on one of those headers in a response
fixture cannot be cleanly localized to "the response builder" —
the request-side fixtures isolate the request builder, but
there's no symmetric way to isolate the response builder for
those copied headers.

We accept this. The alternative is adding `via()`/`from()`/`to()`
setters to `SipResponseBuilder`, which is a public-API change
unrelated to the oracle. The fixture set includes one minimal
"status-only" response (no `from_request`) so the response build
path itself is at least exercised in isolation.

### File layout

One new file: `crates/rsiprtp/tests/builder_roundtrip.rs`. No
shared `_oracle/mod.rs` because there is no fuzz target to share
with.

Internal shape (minimal helper, asserts inline):

```rust
//! Builder typed round-trip oracle.
//! Build typed → serialize → parse → read typed → assert agreement.

use rsiprtp::sip::message::{Method, SipMessage, SipRequest, SipResponse};

fn roundtrip_request(req: SipRequest) -> SipRequest {
    let bytes = req.to_bytes();
    match SipMessage::parse(&bytes).unwrap_or_else(|e| {
        panic!(
            "builder emitted bytes the parser rejects: {e:?}\nbytes:\n{}",
            String::from_utf8_lossy(&bytes),
        )
    }) {
        SipMessage::Request(r) => r,
        SipMessage::Response(_) => panic!("expected Request, got Response"),
    }
}

fn roundtrip_response(resp: SipResponse) -> SipResponse { /* mirror */ }
```

Each fixture builds, calls `roundtrip_*`, and asserts inline:

```rust
#[test]
fn request_invite_minimal() {
    let req = SipRequest::builder()
        .method(Method::Invite)
        .uri("sip:bob@example.com")
        .via("alice.example.com", 5060, "UDP", "z9hG4bKnashds7")
        .from("sip:alice@example.com", "1928301774")
        .to("sip:bob@example.com")
        .call_id("a84b4c76e66710@pc33.example.com")
        .cseq(314159)
        .build().unwrap();
    let r = roundtrip_request(req);
    assert_eq!(r.method(), Method::Invite);
    assert_eq!(r.uri().to_string(), "sip:bob@example.com");
    assert_eq!(r.via_branch().unwrap(), "z9hG4bKnashds7");
    assert_eq!(r.from_tag().unwrap(), "1928301774");
    assert_eq!(r.from_uri().unwrap().to_string(), "sip:alice@example.com");
    assert_eq!(r.to_uri().unwrap().to_string(), "sip:bob@example.com");
    assert_eq!(r.to_tag(), None);
    assert_eq!(r.call_id().unwrap(), "a84b4c76e66710@pc33.example.com");
    assert_eq!(r.cseq().unwrap(), 314159);
    assert_eq!(r.cseq_method().unwrap(), Method::Invite);
}
```

Direct, no hidden Option-checking semantics, failure messages
point straight at the assertion that fired. ~5 LOC of helpers,
~15 LOC per fixture.

### Fixture set

One `#[test]` per fixture. Sized to cover every builder field
that has a setter *and* a typed accessor; each fixture stays
focused on one configuration so a failure points at the
responsible code.

Requests (12):

1. `request_invite_minimal` — method, uri, via, from+tag, to (no
   tag), call-id, cseq, default max-forwards.
2. `request_invite_with_display_name` — adds `from_display`.
   Since there is no typed accessor for display name, the
   assertion is byte-level: `String::from_utf8_lossy(&bytes)
   .contains("\"Alice\"")`. Read-side typed parsing of display
   name is not exercised — see "What the oracle will not catch".
3. `request_ack_with_to_tag` — exercises `to_tag` set.
4. `request_invite_with_body_and_contact` — body + content-type
   + contact URI.
5. `request_register_with_authorization_and_expires` —
   authorization, expires, explicit max_forwards.
6. `request_invite_with_proxy_authorization` — proxy_authorization.
   Separate fixture from #5 so a failure isolates which auth
   header drifted.
7. `request_invite_with_session_expires_and_min_se` — RFC 4028
   typed headers via the `Other` arm.
8. `request_prack_with_rack` — RAck (RFC 3262 §7.2).
9. `request_invite_with_require_supported_allow` — list-typed
   headers (Require, Supported, Allow).
10. `request_bye_with_routes` — multiple Route headers in
    input order (`route_headers()` returns them in iteration
    order over `inner.headers`, which is insertion order —
    `message.rs:438`). BYE method also exercises the CSeq
    method round-trip on a non-INVITE.
11. `request_invite_with_ipv6_via_host` — Via host
    `"[2001:db8::1]"`. Asserts the round-trip preserves the
    bracketed host. Catches any builder regression that
    mis-handles colons in host.
12. `request_options_with_explicit_max_forwards_and_tcp_via` —
    non-default Via transport (`"TCP"`) and `max_forwards(15)`.
    Catches drift in non-default branches of the builder.
    OPTIONS method also exercises the CSeq method round-trip
    on a non-INVITE.

Responses (5):

13. `response_200_ok_minimal` — `.status(200, "OK")` only, no
    `from_request`. Isolates the response build path from the
    request-builder + copy path, which the other response
    fixtures inevitably bundle.
14. `response_200_ok_from_invite` — `from_request(&req)` then
    `to_tag`. Asserts Via/From/To/Call-ID/CSeq round-trip
    through the copy path.
15. `response_200_with_body` — response body + content-type
    (e.g., 200 OK with an SDP answer).
16. `response_200_with_session_expires_and_allow` — RFC 4028
    + Allow on responses.
17. `response_183_with_rseq` — RFC 3262 reliable provisional.

Total: 17 fixtures + ~5 LOC of helpers ≈ 220–250 LOC.

### Skip / pin mechanism

**None at start.** Same reasoning as the M11 and SDP oracles —
this is self-consistency on code we own. If a defensible
non-round-tripping case turns up that we choose not to fix, add a
pinned skip then. Reactively, not up front.

## Trade-offs and acknowledged risks

**Trade-off rejected: a fuzz target.** Generating arbitrary valid
typed inputs requires `Arbitrary` impls for `SipRequestBuilder`'s
26 fields, each with its own validity constraint (URI shape, tag
charset, RAck method, branch token, etc.). The setup cost is
significant, the per-input ROI is low — most arbitrary builder
configurations are uninteresting variants of the same field
formatting. The bytes-in fuzz target already covers the parse
side; this oracle's value is in **shaped** typed inputs, where
~10 hand-picked fixtures hit every code path. Skipping fuzz keeps
this change at ~150 LOC.

**Trade-off rejected: operate at the parser-layer struct
(`parser::Request`) instead of the builder layer.** A `Request`
struct round-trip (`parser_req.to_bytes() → parse → struct-eq`)
is a strictly weaker version of the M11 bytes-in invariant — same
serializer, same parser, same equality. The whole point of the
"other direction" framing is to exercise the *builder*'s
formatting choices and the *typed accessors*' parsing choices,
which the parser-layer struct sidesteps. Builder layer it is.

**Trade-off rejected: assert raw bytes equality at the wire form.**
The bytes are a function of the builder's emit order, which is an
implementation detail. Asserting on the typed accessor's return
value isolates the test from header-emit reordering. (M11 already
guards bytes-equality on parse → serialize round-trips; that's
where it belongs.)

**Risk: the oracle finds nothing.** Same risk as M11 and SDP —
acceptable. Regression-prevention is the long tail of the value.

**Risk: fixtures coincidentally hit "easy" cases that don't
exercise the builder code that's most likely to drift.** Mitigated
by the fixture-per-field rule: every `SipRequestBuilder` /
`SipResponseBuilder` setter that has a corresponding accessor on
the reparsed message gets at least one fixture.

**Risk: fixture maintenance.** When a new field is added to
either builder, the fixture set should grow. We rely on review
discipline for this; there is no CI-level check that "every
builder field has an oracle fixture". A nice-to-have but explicit
out-of-scope here — building that introspection is a much bigger
change than the oracle itself.

## Out of scope

- Fuzz target for typed inputs (justified above).
- Grammar-conformance oracle (catches symmetric loss).
- Changes to `SipRequestBuilder`, `SipResponseBuilder`, the
  parser, or the typed accessors.
- A coverage check that every builder field is fixture-covered.

## Implementation plan

1. Write `crates/rsiprtp/tests/builder_roundtrip.rs` with the
   helpers and 17 fixtures (~220–250 LOC).
2. `cargo test -p rsiprtp --test builder_roundtrip` — confirm the
   suite passes.
3. `cargo clippy --workspace -- -D warnings` and
   `cargo fmt --check` — confirm clean.
4. If a fixture fails, triage like an M11 fuzz finding: identify
   whether the bug is in the builder, the parser-typed layer, or
   the fixture's expectation (canonicalization precondition
   violated?), and fix or pin accordingly.

Total estimated diff: ~220–250 LOC in 1 new file. No changes to
existing code.

## Open questions

None at design stage. Two minor decisions resolved during the
update:

- Display-name round-trip: byte-level assertion only. Adding a
  typed accessor is out of scope for a test-only change.
- Via host/port/transport: assert against the raw Via string
  (via `via_headers_raw()`) since `via_branch()` is the only
  typed accessor. The IPv6 fixture uses raw-string inspection
  for the host check.
