## Context

Track B of the proptest workstream covers the typed SIP headers and the SipUri. The libfuzzer per-header crash targets in `fuzz/fuzz_targets/sip_*` (`sip_via_typed`, `sip_contact`, `sip_name_addr`, `sip_cseq`, `sip_uri`) saturate on bit-mutation. They almost never produce an input that gets past the leading byte without erroring out, because the surface is grammar-gated (`sip:` prefix, `<`/`>` framing, `SIP/2.0/UDP `, `branch=z9hG4bK*`, etc.). What survives the framing dies on the host-or-port shape. As a result, every parser path past the first reject is reached only by serendipity.

A proptest generator that emits **structurally-valid** header values exercises those deeper paths on every iteration. Combined with shrinking, when a property fails the harness automatically minimizes to the smallest input that reproduces the bug — which libfuzzer cannot do for grammar-gated inputs (its byte-level shrinker breaks the grammar before it shrinks anything interesting).

This is **strictly additive** to fuzzing, not a replacement. Fuzzing finds invalid-input crashes; proptest finds valid-input semantic divergences (parse → serialize → parse drift). Both matter.

## Goals

1. Generate structurally-valid wire-form values for `Via`, `From`, `To`, `Contact`, `CSeq`, and `SipUri`.
2. Parse with the in-tree typed parsers (`crate::sip::parser::typed::*`, `crate::sip::uri::SipUri`).
3. Assert a per-header **oracle** — primarily round-trip equivalence of the parsed AST under serialize-then-reparse.
4. Use proptest's shrinking so a failing property minimizes to a small reproducer.
5. Land as a single `tests/` integration test file plus an optional shared helper module — no production-code or `Cargo.toml` changes beyond a single `proptest` dev-dep line (which Track A may also add; both adding it is fine, only one will land — see *File layout*).

## Non-goals (anti-scope)

- **`Authorization` / `WWW-Authenticate` / `Proxy-Authenticate`.** Challenge/response (digest, qop, opaque, nonce) has its own grammar and is out of scope here. Follow-up.
- **`Route` / `Record-Route` lists.** These are typed only at the higher message layer (`SipRequest::route_headers()`), not in `parser/typed/*`. No typed parser to property-test.
- **Headers without a `parse` function in `parser::typed/`.** `Allow`, `Supported`, `Require`, `Session-Expires`, `Min-SE`, `RAck` are formatted/consumed at the `message.rs` builder/accessor layer; that's Track A's territory once Track A's HLD lands.
- **Differential testing against `rsip 0.4`.** Out of scope. The bytes-in oracle (`tests/parser_diff.rs`) covers that pinning. Adding it here would couple this work to `rsip` shape.
- **Negative-input properties** ("invalid input always errors gracefully"). The libfuzzer crash targets already do this. Properties here generate *valid* inputs.
- **CSeq method coverage of unknown extension methods.** The typed `Method` enum is closed (14 variants); the parser rejects unknowns. Generators draw from the closed set.

## What's already in place

Per-header `parse` functions and `Display` impls — every typed header in scope has both, so the round-trip oracle is `parse → to_string → parse → AST equality`:

| Header | Parse fn | Serialize | Notes |
|--------|----------|-----------|-------|
| `Via` | `Via::parse(&str)` (`typed/via.rs:62`) | `impl Display` (`via.rs:187`) | params order preserved verbatim; tri-state rport |
| `From` | `From::parse(&str)` (`typed/from.rs:39`) | `impl Display` via `NameAddr` (`from.rs:58`) | delegates to `NameAddr::parse` |
| `To` | `To::parse(&str)` (`typed/to.rs:30`) | `impl Display` via `NameAddr` (`to.rs:49`) | delegates to `NameAddr::parse` |
| `Contact` | `Contact::parse(&str)` (`typed/contact.rs:113`) | `impl Display` (`contact.rs:170`) | enum: `Wildcard{params}` or `Addr(ContactAddr)` |
| `CSeq` | `CSeq::parse(&str)` (`typed/cseq.rs:36`) | `impl Display` (`cseq.rs:58`) | canonicalizes method to upper-case |
| `SipUri` | `SipUri::parse(&str)` (`uri.rs:105`) | `impl Display` (`uri.rs:375`) | scheme normalized to lowercase |
| `NameAddr` | `NameAddr::parse(&str)` (`name_addr.rs:65`) | `impl Display` (`name_addr.rs:150`) | always emits `<...>` even for bare addr-spec input |

**Important asymmetry to call out** — `NameAddr::Display` always emits angle brackets, even for a bare addr-spec on input. So a generator that emits `sip:bob@example.com` (bare) without angle brackets will round-trip to `<sip:bob@example.com>`, *not* the original string. The round-trip oracle must compare **parsed structs**, not raw wire strings, to be sound. This is fine — it's the standard parse-serialize-reparse fixed-point shape, not string equality.

Same caveat applies to `SipUri` more subtly: the parser is lenient (silently drops empty params after `;;`, accepts trailing `;`, etc.), so wire-string equality is wrong; AST equality after one round of normalization is the contract.

## Per-header coverage matrix

| Header | Parse fn | Serialize fn | Oracle | Generator strategy summary |
|--------|----------|--------------|--------|---------------------------|
| `SipUri` | `SipUri::parse` | `Display` | `parse(s) == parse(s.to_string())` after first normalization round | scheme {sip, sips} (defer tel to v2 — different shape); optional userinfo from token-set chars; host from a small fixed set of hostnames + IPv4 octet quads + bracketed IPv6; optional u16 port; flag/value params drawn from `{transport, lr, maddr, ttl, branch}`; URI headers limited to ≤4 entries with safe key/value chars |
| `Via` | `Via::parse` | `Display` | `parse(s) == parse(s.to_string())`; structural assertions: branch starts with `z9hG4bK`; rport tri-state survives | protocol = `"SIP"`, version = `"2.0"`; transport ∈ {UDP, TCP, TLS, WS, WSS}; sent-by from a hostname/IPv4/bracketed-IPv6 generator + optional port; params: branch (mandatory in v1, prefix-locked); rport flag-or-value; received (IP); maddr (IP); ttl (u8) |
| `From` | `From::parse` | `Display` via `NameAddr` | `parse(s) == parse(s.to_string())` | optional display name (token vs. quoted); SipUri generator (from above); optional `tag` parameter from token chars |
| `To` | `To::parse` | `Display` via `NameAddr` | same as From | same as From, but `tag` is optional and frequently absent (initial-INVITE shape) |
| `Contact` | `Contact::parse` | `Display` | `parse(s) == parse(s.to_string())`; if wildcard, `is_wildcard()` holds; if Addr, `expires()`/`q_value()` recover the typed value | enum mix: 80 % `Addr` (NameAddr-shaped, plus optional `expires` u32 and `q` ∈ [0.0, 1.0] with one-decimal-place format), 20 % `Wildcard` (bare `*` or `*;expires=0`) |
| `CSeq` | `CSeq::parse` | `Display` | `parse(s).seq == seq && parse(s).method == method`; round-trip via `parse(c.to_string())` | seq: `any::<u32>()`; method: drawn from the 14-variant `Method` enum |
| `NameAddr` | `NameAddr::parse` | `Display` | parse-serialize-reparse fixed point | shared internal generator; not directly asserted as a top-level property (covered indirectly by From/To/Contact). **Only** add a separate property if a From/To/Contact failure points to a NameAddr-only path (avoid double-counting). |

## Strategy tree per header

All sketches use real proptest 1.x syntax. `prop_compose!` defines composable strategies; `proptest!` blocks define properties. We deliberately bias toward strategies that stay in the valid set under shrinking — proptest's default shrinker walks toward the empty/zero element, which is *not* a valid Via, so we use `prop_compose!` with constrained sub-strategies instead of `prop_filter` where possible.

### Shared helpers (host, token, IPv4, IPv6)

```rust
// tests/proptest_sip_typed_common/mod.rs (or inline in proptest_sip_typed.rs)

use proptest::prelude::*;

/// RFC 3261 §25.1 token chars: alphanumeric plus `- . ! % * _ + ` ' ~`.
/// Lowercase only in v1 to keep param-key matching identity-clean
/// (parser is case-insensitive on lookups but case-preserving on
/// storage; generators stay lowercase to avoid spurious diffs).
pub fn token_char() -> impl Strategy<Value = char> {
    prop_oneof![
        9 => prop::char::range('a', 'z'),
        9 => prop::char::range('0', '9'),
        1 => Just('-'),
        1 => Just('.'),
        1 => Just('_'),
    ]
}

/// 1..=16 token chars.
pub fn token() -> impl Strategy<Value = String> {
    prop::collection::vec(token_char(), 1..16)
        .prop_map(|cs| cs.into_iter().collect())
}

/// Hostname-shaped: 1..=3 dot-separated labels of token chars.
/// Excludes leading/trailing dots; that's the parser-rejecting shape.
pub fn hostname() -> impl Strategy<Value = String> {
    prop::collection::vec(token(), 1..=3)
        .prop_map(|labels| labels.join("."))
}

/// IPv4 dotted quad.
pub fn ipv4() -> impl Strategy<Value = String> {
    (any::<u8>(), any::<u8>(), any::<u8>(), any::<u8>())
        .prop_map(|(a, b, c, d)| format!("{a}.{b}.{c}.{d}"))
}

/// Bracketed IPv6 — small fixed set in v1 (full IPv6 generation is its
/// own beast and not load-bearing for typed-header round-trip; add in v2
/// if signal warrants).
pub fn ipv6_bracketed() -> impl Strategy<Value = String> {
    prop_oneof![
        Just("[::1]".to_string()),
        Just("[2001:db8::1]".to_string()),
        Just("[fe80::1]".to_string()),
    ]
}

/// host = hostname | ipv4 | bracketed IPv6.
pub fn host() -> impl Strategy<Value = String> {
    prop_oneof![
        6 => hostname(),
        3 => ipv4(),
        1 => ipv6_bracketed(),
    ]
}

/// Optional port — any u16 (including 0; parser accepts).
pub fn opt_port() -> impl Strategy<Value = Option<u16>> {
    prop_oneof![
        1 => Just(None),
        2 => any::<u16>().prop_map(Some),
    ]
}

/// host[:port] — emitted with the colon when the port is present.
/// Bracketed IPv6 already has its `]`, port comes after.
pub fn host_port() -> impl Strategy<Value = String> {
    (host(), opt_port()).prop_map(|(h, p)| match p {
        None => h,
        Some(port) => format!("{h}:{port}"),
    })
}
```

### SipUri

```rust
mod sip_uri {
    use super::common::*;
    use proptest::prelude::*;
    use rsiprtp::sip::SipUri;

    /// One URI parameter: a (key, Option<value>) pair.
    /// Keys are tokens; values are also tokens. We exclude `;`, `?`, and
    /// `=` from values in v1 because the URI parser splits on those.
    /// Quoted values are out of scope for v1 — the URI grammar (RFC 3261
    /// §19.1) doesn't permit them; quoted values live on the *header*
    /// parameters (NameAddr layer).
    fn uri_param() -> impl Strategy<Value = (String, Option<String>)> {
        (token(), prop::option::of(token()))
    }

    /// 0..=4 URI params. We don't test the 32-cap here (that's a fuzz
    /// concern); 0..=4 stays well inside the fixed-point oracle.
    fn uri_params() -> impl Strategy<Value = Vec<(String, Option<String>)>> {
        prop::collection::vec(uri_param(), 0..=4)
    }

    /// One URI header: (key, value). Values restricted to token chars
    /// because `?` and `&` are the URI-header separators.
    fn uri_header() -> impl Strategy<Value = (String, String)> {
        (token(), token())
    }

    /// 0..=3 URI headers. The parser caps at 32 — well above.
    fn uri_headers() -> impl Strategy<Value = Vec<(String, String)>> {
        prop::collection::vec(uri_header(), 0..=3)
    }

    /// Build a wire-form SIP URI string from the component parts.
    /// We bypass the in-tree builder to keep generator and parser
    /// independent — a builder bug would falsely silence a parser bug.
    prop_compose! {
        fn sip_uri_wire()(
            scheme in prop_oneof![Just("sip"), Just("sips")],
            user in prop::option::of(token()),
            host in host(),
            port in opt_port(),
            params in uri_params(),
            headers in uri_headers(),
        ) -> String {
            let mut s = String::new();
            s.push_str(scheme);
            s.push(':');
            if let Some(u) = &user {
                s.push_str(u);
                s.push('@');
            }
            s.push_str(&host);
            if let Some(p) = port {
                s.push_str(&format!(":{p}"));
            }
            for (k, v) in &params {
                s.push(';');
                s.push_str(k);
                if let Some(val) = v {
                    s.push('=');
                    s.push_str(val);
                }
            }
            if !headers.is_empty() {
                s.push('?');
                for (i, (k, v)) in headers.iter().enumerate() {
                    if i > 0 { s.push('&'); }
                    s.push_str(k);
                    s.push('=');
                    s.push_str(v);
                }
            }
            s
        }
    }

    proptest! {
        #![proptest_config(ProptestConfig::with_cases(256))]

        /// Round-trip fixed point: parse, serialize, reparse, and the
        /// two parses must be AST-equal. This is the standard
        /// `assert_roundtrip_fixed_point` shape, scoped to SipUri.
        #[test]
        fn sip_uri_round_trip(s in sip_uri_wire()) {
            let u1 = SipUri::parse(&s).expect("generator must produce parseable URI");
            let s2 = u1.to_string();
            let u2 = SipUri::parse(&s2).expect("serialized form must reparse");
            prop_assert_eq!(u1, u2);
        }
    }
}
```

**Sub-strategies excluded in v1** (flagged for follow-up if signal warrants):
- `tel:` URI scheme (different shape; `host` is a phone-number, no userinfo, no port).
- Percent-encoding in userinfo (`sip:user%40foo@host`). The parser does not decode; identity holds, but the generator would have to enforce `%xx` shape and skip valid sequences that decode to bytes the parser would re-encode differently.
- IPv6 zone-id (`[fe80::1%eth0]`). Niche, defer.
- Empty / pathological params (`sip:host;;` — parser is lenient, but the generator-shrinker pair can't reliably keep this in the valid set).
- `password` after the user (`user:pass@host`). Parser folds it into `user`; round-trip is fine but the structural assertion is muddier.

### Via

```rust
mod via {
    use super::common::*;
    use proptest::prelude::*;
    use rsiprtp::sip::parser::typed::Via;

    /// Branch parameter value. RFC 3261 §8.1.1.7 mandates `z9hG4bK`
    /// prefix for an RFC 3261 transaction. Our parser does NOT enforce
    /// this (transaction layer's job per `via.rs` doc comment), so a
    /// generator that drops it would still produce parseable Vias —
    /// but enforcing the prefix here documents the canonical shape and
    /// keeps shrinking inside the standards-compliant set.
    fn branch_value() -> impl Strategy<Value = String> {
        token().prop_map(|suffix| format!("z9hG4bK{suffix}"))
    }

    /// rport: tri-state — absent, flag, or value.
    fn rport_param() -> impl Strategy<Value = Option<(String, Option<String>)>> {
        prop_oneof![
            2 => Just(None),
            1 => Just(Some(("rport".to_string(), None))),
            1 => any::<u16>()
                .prop_map(|p| Some(("rport".to_string(), Some(p.to_string())))),
        ]
    }

    /// Optional received= (always an IPv4 literal in v1).
    fn received_param() -> impl Strategy<Value = Option<(String, Option<String>)>> {
        prop_oneof![
            2 => Just(None),
            1 => ipv4().prop_map(|ip| Some(("received".to_string(), Some(ip)))),
        ]
    }

    /// Optional ttl=N (u8).
    fn ttl_param() -> impl Strategy<Value = Option<(String, Option<String>)>> {
        prop_oneof![
            3 => Just(None),
            1 => any::<u8>()
                .prop_map(|t| Some(("ttl".to_string(), Some(t.to_string())))),
        ]
    }

    /// Optional maddr= (IPv4 in v1).
    fn maddr_param() -> impl Strategy<Value = Option<(String, Option<String>)>> {
        prop_oneof![
            3 => Just(None),
            1 => ipv4().prop_map(|m| Some(("maddr".to_string(), Some(m)))),
        ]
    }

    prop_compose! {
        fn via_wire()(
            transport in prop_oneof![
                Just("UDP"), Just("TCP"), Just("TLS"),
                Just("WS"), Just("WSS"),
            ],
            sent_by in host_port(),
            branch in branch_value(),
            rport in rport_param(),
            received in received_param(),
            ttl in ttl_param(),
            maddr in maddr_param(),
        ) -> String {
            let mut s = format!("SIP/2.0/{transport} {sent_by};branch={branch}");
            for opt in [received, rport, ttl, maddr] {
                if let Some((k, v)) = opt {
                    match v {
                        Some(val) => s.push_str(&format!(";{k}={val}")),
                        None => s.push_str(&format!(";{k}")),
                    }
                }
            }
            s
        }
    }

    proptest! {
        #![proptest_config(ProptestConfig::with_cases(256))]

        #[test]
        fn via_round_trip(s in via_wire()) {
            let v1 = Via::parse(&s).expect("generator must produce parseable Via");
            let s2 = v1.to_string();
            let v2 = Via::parse(&s2).expect("serialized Via must reparse");
            prop_assert_eq!(v1, v2);
        }

        #[test]
        fn via_branch_starts_with_cookie(s in via_wire()) {
            let v = Via::parse(&s).unwrap();
            let b = v.branch().expect("generator always emits branch");
            prop_assert!(b.starts_with("z9hG4bK"));
        }

        #[test]
        fn via_rport_tri_state_round_trips(s in via_wire()) {
            let v1 = Via::parse(&s).unwrap();
            let v2 = Via::parse(&v1.to_string()).unwrap();
            prop_assert_eq!(v1.rport(), v2.rport());
        }
    }
}
```

**v1 exclusions:**
- Multi-`Via` comma-joined values. The typed parser intentionally does *not* split on `,` (see `via.rs:38`); that's the framing layer's concern. One via-parm per generated string.
- Custom transport tokens (e.g. `SCTP`, `QUIC`). Parser accepts case-preserved, but enumerating an open set without false-positive risk is awkward. Closed set above is enough.
- Embedded quoted-string param values (e.g. `;branch="z9hG4bK foo"`). RFC 3261 grammar permits `gen-value = token / host / quoted-string`, but the typed Via parser handles quoting internally via `parse_params`; generator complexity not justified in v1.
- Capital-letter param keys. Parser lookup is case-insensitive, but Display preserves case, so a key generated as `BRANCH` would round-trip as `BRANCH` — fine for the `==` oracle, but adds noise without coverage value.

### From / To (NameAddr-backed)

```rust
mod name_addr_based {
    use super::common::*;
    use super::sip_uri::sip_uri_wire;
    use proptest::prelude::*;
    use rsiprtp::sip::parser::typed::{From as TypedFrom, To as TypedTo};

    /// Display-name strategies. Two shapes:
    /// - bare token sequence: a single token (no quoting needed)
    /// - quoted: arbitrary printable ASCII excluding `"` and `\` (we'd
    ///   need to escape those, and proptest's String shrinker is
    ///   notoriously poor at preserving escape correctness — easier to
    ///   exclude them in v1)
    fn display_name_token() -> impl Strategy<Value = String> {
        token()
    }

    fn display_name_quoted_inner() -> impl Strategy<Value = String> {
        // Printable ASCII minus `"` (0x22) and `\` (0x5C).
        // Length 1..=20.
        prop::collection::vec(
            prop::char::range(' ', '~')
                .prop_filter("no quote/backslash",
                    |c| *c != '"' && *c != '\\'),
            1..=20,
        ).prop_map(|cs| cs.into_iter().collect())
    }

    /// Either form, prefixed for emission. The "" Display path is what
    /// the parser actually sees.
    fn display_name_emitted() -> impl Strategy<Value = String> {
        prop_oneof![
            // bare token
            display_name_token().prop_map(|t| format!("{t} ")),
            // quoted
            display_name_quoted_inner().prop_map(|q| format!("\"{q}\" ")),
            // absent
            Just(String::new()),
        ]
    }

    prop_compose! {
        fn from_wire()(
            display in display_name_emitted(),
            uri in sip_uri_wire(),
            tag_present in any::<bool>(),
            tag_value in token(),
        ) -> String {
            let bracketed = !display.is_empty();  // forced
            let core = if bracketed {
                format!("{display}<{uri}>")
            } else {
                format!("{display}{uri}")
            };
            if tag_present {
                format!("{core};tag={tag_value}")
            } else {
                core
            }
        }
    }

    proptest! {
        #![proptest_config(ProptestConfig::with_cases(256))]

        #[test]
        fn from_round_trip(s in from_wire()) {
            let f1 = TypedFrom::parse(&s).expect("generator must produce parseable From");
            let s2 = f1.to_string();
            let f2 = TypedFrom::parse(&s2).expect("From must reparse");
            prop_assert_eq!(f1, f2);
        }

        /// To is structurally identical to From; reuse the same generator.
        #[test]
        fn to_round_trip(s in from_wire()) {
            let t1 = TypedTo::parse(&s).expect("generator must produce parseable To");
            let s2 = t1.to_string();
            let t2 = TypedTo::parse(&s2).expect("To must reparse");
            prop_assert_eq!(t1, t2);
        }
    }
}
```

**v1 exclusions:**
- Display-name characters that require quoted-pair escaping (`\"`, `\\`). The escape round-trip is correct in the parser (verified by `test_quoted_display_name_with_escapes`) but the proptest generator would need to track escape boundaries through shrinking. Defer.
- UTF-8 multi-byte display names. Parser handles them (see `quoted_display_preserves_utf8_multibyte`), but proptest's `prop::char::range` over the full Unicode range produces shrinking storms that aren't load-bearing for typed-header bugs. Restrict to ASCII printable in v1.
- Multiple parameters beyond `tag`. `From` legitimately permits arbitrary `;k=v` parameters per RFC 3261 §25.1; we restrict to `tag` to keep the generator focused. Add `;<token>=<token>` extension parameters in v2 if useful.

### Contact

```rust
mod contact {
    use super::common::*;
    use super::name_addr_based::from_wire;
    use proptest::prelude::*;
    use rsiprtp::sip::parser::typed::Contact;

    /// q-value with one decimal place: 0.0, 0.1, …, 1.0. Parser accepts
    /// any f32-parseable string but exact float round-trip via Display
    /// is fragile; one-decimal-place keeps the AST equality safe.
    fn q_value_str() -> impl Strategy<Value = String> {
        (0u8..=10).prop_map(|n| {
            if n == 10 { "1.0".to_string() }
            else { format!("0.{n}") }
        })
    }

    /// Wildcard form: bare `*` or `*;expires=NN` or `*;expires=NN;extra`.
    fn wildcard_wire() -> impl Strategy<Value = String> {
        prop_oneof![
            Just("*".to_string()),
            any::<u32>().prop_map(|e| format!("*;expires={e}")),
        ]
    }

    /// Addr form: a name-address (reuse from_wire's URI/display-name
    /// machinery, conceptually; or copy the prop_compose for clarity)
    /// plus optional expires + q-value tail.
    prop_compose! {
        fn addr_wire()(
            base in from_wire(),    // URI + optional display + optional tag
            expires in prop::option::of(any::<u32>()),
            q in prop::option::of(q_value_str()),
        ) -> String {
            let mut s = base;  // already has any tag already appended
            if let Some(e) = expires {
                s.push_str(&format!(";expires={e}"));
            }
            if let Some(q) = q {
                s.push_str(&format!(";q={q}"));
            }
            s
        }
    }

    fn contact_wire() -> impl Strategy<Value = String> {
        prop_oneof![
            8 => addr_wire(),
            2 => wildcard_wire(),
        ]
    }

    proptest! {
        #![proptest_config(ProptestConfig::with_cases(256))]

        #[test]
        fn contact_round_trip(s in contact_wire()) {
            let c1 = Contact::parse(&s).expect("generator must produce parseable Contact");
            let s2 = c1.to_string();
            let c2 = Contact::parse(&s2).expect("Contact must reparse");
            prop_assert_eq!(c1, c2);
        }

        /// Wildcard discriminator preserved through round-trip.
        #[test]
        fn contact_wildcard_preserved(s in wildcard_wire()) {
            let c = Contact::parse(&s).unwrap();
            prop_assert!(c.is_wildcard());
            let c2 = Contact::parse(&c.to_string()).unwrap();
            prop_assert!(c2.is_wildcard());
        }

        /// expires accessor recovers the typed u32 (when present).
        #[test]
        fn contact_expires_round_trips(e in any::<u32>()) {
            let s = format!("*;expires={e}");
            let c = Contact::parse(&s).unwrap();
            prop_assert_eq!(c.expires(), Some(e));
        }
    }
}
```

**v1 exclusions:**
- Comma-separated multiple Contacts (same rationale as Via — typed parser doesn't split).
- Contact with wildcard + display name. Wildcard-with-display-name is grammatically excluded by RFC 3261 §20.10 ABNF (`STAR` and `contact-param` are alternation branches).
- `q` values with more than one decimal place (`0.55`, `0.123`). Parser accepts; AST equality on `f32` rounding is brittle.

### CSeq

```rust
mod cseq {
    use proptest::prelude::*;
    use rsiprtp::sip::parser::method::Method;
    use rsiprtp::sip::parser::typed::CSeq;

    fn method_strategy() -> impl Strategy<Value = Method> {
        prop_oneof![
            Just(Method::Invite),  Just(Method::Ack),     Just(Method::Bye),
            Just(Method::Cancel),  Just(Method::Register),Just(Method::Options),
            Just(Method::Info),    Just(Method::Update),  Just(Method::Refer),
            Just(Method::Notify),  Just(Method::Subscribe),Just(Method::Prack),
            Just(Method::Message), Just(Method::Publish),
        ]
    }

    prop_compose! {
        fn cseq_pair()(
            seq in any::<u32>(),
            method in method_strategy(),
        ) -> (u32, Method) {
            (seq, method)
        }
    }

    proptest! {
        #![proptest_config(ProptestConfig::with_cases(256))]

        /// AST equivalence: parse the wire form, the resulting CSeq's
        /// fields must equal the inputs.
        #[test]
        fn cseq_parse_recovers_inputs((seq, method) in cseq_pair()) {
            let s = format!("{seq} {method}");
            let c = CSeq::parse(&s).expect("CSeq must parse");
            prop_assert_eq!(c.seq, seq);
            prop_assert_eq!(c.method, method);
        }

        /// Round-trip via Display (which canonicalizes method case).
        #[test]
        fn cseq_round_trip((seq, method) in cseq_pair()) {
            let s = format!("{seq} {method}");
            let c1 = CSeq::parse(&s).unwrap();
            let s2 = c1.to_string();
            let c2 = CSeq::parse(&s2).unwrap();
            prop_assert_eq!(c1, c2);
        }

        /// Method case-insensitivity: lowercase method on input still
        /// recovers the Method enum, and round-trip canonicalizes to
        /// upper-case (so reparse equality still holds).
        #[test]
        fn cseq_method_case_insensitive((seq, method) in cseq_pair()) {
            let lower = method.to_string().to_ascii_lowercase();
            let s = format!("{seq} {lower}");
            let c = CSeq::parse(&s).unwrap();
            prop_assert_eq!(c.method, method);
        }
    }
}
```

CSeq has the smallest state space — `u32 × Method` — and the most-direct oracle (the typed fields directly recoverable). 256 cases is generous.

## Oracles

For every header in scope, the oracle is **parse-serialize-reparse fixed point on the AST**:

```text
let a1 = Header::parse(s)?;
let s2 = a1.to_string();
let a2 = Header::parse(&s2)?;
assert_eq!(a1, a2);
```

This is the same shape as `parser_roundtrip_oracle::assert_roundtrip_fixed_point` for whole messages (`crates/rsiprtp/tests/parser_roundtrip_oracle/mod.rs`), scoped to a header. Wire-string equality (`s == s2`) is **deliberately not** asserted, because the parsers are lenient (multiple `;;`, trailing `;`, bare addr-spec → bracketed, scheme case normalization, lowercase method canonicalization) — wire equality would falsely fail. AST equality after the first round of normalization is the contract.

Per-header structural assertions are added on top where they differentiate well:

| Header | Extra structural assertion |
|--------|----------------------------|
| `Via` | `branch().is_some()`, `branch().unwrap().starts_with("z9hG4bK")`, `rport()` tri-state preserved |
| `From` | (none — round-trip alone) |
| `To` | (none) |
| `Contact` | `is_wildcard()` discriminator preserved; `expires()` and `q_value()` recover typed values when emitted |
| `CSeq` | `c.seq == input_seq`, `c.method == input_method` |
| `SipUri` | `scheme_enum()`, `user()`, `host()`, `port()` all preserved |

No header in scope is parse-only — every one has a `Display` impl. The HLD-flagged "if no Display, parse-only oracle" caveat does **not** apply to v1.

## Shrinking

Proptest's shrinker walks toward the empty/zero element of each strategy. For grammar-bound generators this can shrink the input *out of the valid set* unless the generator is composed of strategies whose individual shrinks each stay valid.

**Risk inventory — failure modes and mitigations:**

1. **Via branch shrinking.** A naive `branch in token()` that emits the prefix as part of the value would let proptest shrink the branch to `""` or `"a"`, which the parser accepts (the `z9hG4bK` cookie is RFC-mandated but our parser doesn't enforce — see `via.rs:137` doc comment). That's still a valid Via, but the structural assertion `branch().unwrap().starts_with("z9hG4bK")` would fail on the shrunk input — masking the real bug.
   **Mitigation:** the strategy `branch_value()` uses `token().prop_map(|s| format!("z9hG4bK{s}"))` so the prefix is *outside the shrink space*. Only the suffix shrinks; the cookie always stays.

2. **Display name → `<addr-spec>` boundary in NameAddr.** If the generator independently picks "display-name present" and "use brackets", the shrinker can shrink one without the other and produce `Alice sip:a@b` (no brackets, no display) which the parser rejects.
   **Mitigation:** inside `prop_compose!`, force `bracketed = !display.is_empty()` rather than offering it as an independent strategy axis. The shrinker can shrink `display` to empty (which un-brackets cleanly) but cannot produce the invalid combination.

3. **`prop_filter` antipattern.** Using `prop_filter("…", |s| Header::parse(s).is_ok())` looks tempting: just filter out anything the parser rejects. It's a trap — the shrinker repeatedly produces invalid candidates and proptest counts those toward the rejection budget (default 65536), eventually erroring with `Reason: too many global rejects`. Even when it doesn't error, the shrinker's local minimization is broken because each step has a high probability of being rejected.
   **Mitigation:** use composite generators that cannot produce invalid output, never filter. The strategies above are constructed accordingly.

4. **Float formatting in Contact.q.** Generating `q` from `any::<f32>()` and formatting via `format!("{q}")` produces non-round-trippable values like `0.30000001`. Reparsing-to-f32 is value-equal but Display would emit `0.30000001` and parse it back as the same f32 — fine for AST equality, but proptest's shrinker will land on hostile float patterns (NaN, infinity, `-0.0`) the parser rejects.
   **Mitigation:** `q_value_str()` strategy returns a pre-formatted string from the closed set `{0.0, 0.1, …, 1.0}`. Shrinking moves toward `0.0`, which is always parseable.

5. **u32 seq generation in CSeq.** `any::<u32>()` includes `0`, which the parser accepts (see `test_parse_zero` in `cseq.rs`). No mitigation needed — full-range generation is correct.

6. **Hostname generation shrinking to empty label.** `prop::collection::vec(token(), 1..=3)` always emits at least one label of at least one char, so `hostname()` cannot shrink to `""`. Joining with `.` cannot produce a leading or trailing dot because each label is non-empty.

The pattern across all of these: **build invariants into the strategy structure, not into post-hoc filters.** When in doubt, lean on `prop_compose!` to encode constraints, not `prop_filter`.

`prop_assume!` has its place for assertions about the *parsed* shape (e.g. "if the generator emitted a tag, the parsed `tag()` must be `Some`"), but never for input shape.

## Run shape

```rust
proptest! {
    #![proptest_config(ProptestConfig::with_cases(256))]
    // … properties …
}
```

Default 256 cases per property is plenty for the grammars in scope:
- `CSeq`: state space is `u32 × 14`; 256 covers ~256/(2^32 × 14) ≈ negligible coverage in pure-numeric terms, but every property here is "does the parser distinguish these cases from each other" — 256 random samples from u32 give well-distributed boundary coverage (`0`, `u32::MAX`, ~6 powers-of-two).
- `Via` / `From` / `To` / `Contact`: combinatorial state across 4–8 axes; 256 hits ~3 unique values per axis on average, plus the shrinker explores adjacent valid points around any failure.
- `SipUri`: largest state space (scheme × user × host × port × params × headers); 256 cases is the floor. **Flag for monitoring:** if real-world usage shows divergences that 256-case runs miss, bump to 1024 or 4096 for `sip_uri_round_trip` only.

Total runtime: ~10–30 seconds for the full file at 256 cases × 10 properties. Well within `cargo test` reasonable bounds. No `--release` requirement.

CI: runs as part of `cargo test -p rsiprtp` like every other integration test. No new CI config.

## Anti-scope (recap)

Out of scope for this HLD:
1. `Authorization` / `WWW-Authenticate` / `Proxy-Authenticate` / `Proxy-Authorization` (digest challenge/response — separate grammar, follow-up).
2. `Route` / `Record-Route` lists (typed only at message layer, not in `parser/typed/*`).
3. Any header without a `parse` function in `parser::typed/` today: `Allow`, `Supported`, `Require`, `Session-Expires`, `Min-SE`, `RAck`, `Reason`, `Diversion`, etc.
4. `tel:` URIs (parser supports them; not load-bearing for SIP-message round-trip — defer to v2).
5. Percent-encoding round-trip in URI userinfo.
6. Comma-separated multi-value header forms (typed parsers don't split).
7. Differential testing against `rsip 0.4` (Track A's domain).
8. Negative-input properties (libfuzzer covers).
9. UTF-8 multi-byte display names (parser supports; generator complexity not justified in v1).
10. Quoted-pair escapes in display names (parser supports; same reason).

## Open questions

1. **(Most important.)** *Are header parameters serialized in a canonical order, or in input wire order?* The Via/Contact/From/NameAddr Display impls iterate `self.params` in storage order, which is **input wire order**. Our generators emit params in a fixed order (branch first, then received/rport/ttl/maddr in a defined sequence); the parser preserves order; Display emits them in that same order. So as long as generator order matches the order the parsed AST stores them in, AST equality holds for the round-trip.
   But: should the generator deliberately *randomize* parameter order (e.g. `branch` not always first)? RFC 3261 doesn't mandate an order. If the parser's AST equality is order-sensitive (it is — `params: Vec<(String, Option<String>)>`), randomizing order means the *input* and *parsed-then-displayed* still round-trip, but *we lose the property test's ability to detect a future bug where the serializer reorders*.
   **Recommendation:** v1 generators emit params in a fixed order (matching what production builders produce). v2 considers a randomized-order property to assert "the parser preserves wire order under round-trip" as a separate property, distinct from the round-trip-fixed-point property. Confirm with supervisor before locking.

2. **Should we commit `proptest-regressions/` files?** Pro: seeds the next run with the previous failing input — guarantees the regression is caught immediately. Con: adds noise to PR diffs; the shrunk failure also lives in the failing test's panic message and a reproducer can be re-derived. Most Rust projects gitignore the file; a few commit it for paranoia. **Recommendation:** gitignore in v1; revisit if we hit a regression that the strategy genuinely re-shrinks differently across runs.

3. **Do we want a shared property `prop_assert_round_trip!` macro?** Across the 6 modules, the round-trip oracle body is 4 lines repeated 6 times. A macro flattens the noise but adds an indirection reviewers have to chase. **Recommendation:** v1 inlines the 4 lines (matches `assert_roundtrip_fixed_point`'s shape). Promote to a macro only if more headers come in scope.

4. **Should NameAddr have its own property in addition to From/To?** From and To are 1:1 wrappers (see `from.rs:39`, `to.rs:30`); their tests cover NameAddr's parse paths transitively. **Recommendation:** no separate NameAddr property in v1 (per-header fuzz HLD already noted ~95 % overlap with Contact). Add one only if a future From/To failure points to a NameAddr-specific path.

5. **Generator key-case coverage.** Parser lookups are case-insensitive on parameter keys (`tag` vs `TAG` vs `Tag` all match). Display preserves storage case. Should generators emit mixed-case keys to exercise this? **Recommendation:** v1 keeps keys lowercase. Round-trip equality holds either way (Display preserves case; reparse stores the same string), so randomizing case adds no oracle-detectable signal — it would only catch a regression where Display *changed* the key case, which is a different property entirely.
