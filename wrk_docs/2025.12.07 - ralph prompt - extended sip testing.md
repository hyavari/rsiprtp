# Extended SIP Testing with Baresip and Asterisk

## Overview

Extend the existing baresip integration test framework to include comprehensive testing against both baresip (as a SIP UA) and Asterisk (as a SIP PBX/registrar). This creates a complete test suite covering real-world VoIP scenarios.

## Goals

1. **Asterisk Integration** - Add Asterisk as a test target for PBX scenarios
2. **Advanced SIP Scenarios** - Call transfers, conferences, forking, authentication
3. **Media Verification** - Actual RTP audio validation, not just signaling
4. **Security Testing** - TLS transport, SRTP media encryption, DTLS-SRTP
5. **Stress Testing** - High call volume, rapid cycling, resource exhaustion
6. **Error Recovery** - Network failures, timeouts, malformed messages

## Implementation Plan

### Phase 1: Asterisk Controller (New)

Create `crates/mdsiprtp/tests/baresip_integration/framework/asterisk.rs`:

```rust
/// Asterisk PBX controller for integration testing.
pub struct AsteriskInstance {
    process: Option<Child>,
    config_dir: PathBuf,
    ami_port: u16,      // Asterisk Manager Interface
    sip_port: u16,
    rtp_port_start: u16,
}

impl AsteriskInstance {
    /// Start Asterisk with test configuration
    pub async fn new(config: AsteriskConfig) -> Result<Self>;

    /// Connect to AMI for control
    pub async fn connect_ami(&mut self) -> Result<()>;

    /// Register an endpoint
    pub async fn expect_registration(&self, user: &str, timeout: Duration) -> Result<()>;

    /// Originate a call via AMI
    pub async fn originate(&self, channel: &str, extension: &str) -> Result<String>;

    /// Get channel status
    pub async fn channel_status(&self, channel: &str) -> Result<ChannelState>;

    /// Hangup a channel
    pub async fn hangup(&self, channel: &str) -> Result<()>;

    /// Bridge two channels (for transfer testing)
    pub async fn bridge(&self, channel1: &str, channel2: &str) -> Result<()>;
}
```

Configuration generation for Asterisk:
- `sip.conf` / `pjsip.conf` - SIP endpoint configuration
- `extensions.conf` - Dialplan for test scenarios
- `manager.conf` - AMI access
- `rtp.conf` - RTP port ranges
