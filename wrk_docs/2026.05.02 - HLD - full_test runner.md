# 2026.05.02 - HLD - full_test runner

## Goal

A single command that runs the whole rsiprtp test bar — formatter,
clippy, unit + integration tests, doc build, coverage — and writes a
self-contained, dated HTML report under `crates/rsiprtp/tests/results/`.

Modeled on `mddosem`'s `full_test` (`d:\language\mddosem\src\bin\full_test.rs`).
Same idea, much smaller surface, because rsiprtp does not have the
modal complexity (interpreter vs JIT, ordinal/litmus checks, screenshot
capture, DOOM benchmark) that `mddosem`'s runner accreted.

## Shape of the change

A new workspace member `tools/full_test/` containing one `main.rs` file.
It shells out to `cargo` with `Command`, parses libtest output, writes
one HTML report, and exits non-zero if anything failed.

```
tools/
  full_test/
    Cargo.toml
    src/
      main.rs
```

`Cargo.toml` workspace `members` gains `tools/full_test`.
`default-members` stays `["crates/rsiprtp"]` so `cargo build` is
unchanged for everyone else; the runner is invoked explicitly with
`cargo run --release -p full_test`.

Reports land at `crates/rsiprtp/tests/results/YYYY.MM.DD - full test report.html`,
mirroring the mddosem layout (date-stamped, `(N)` suffix on collision,
HTML only).

## Why a separate sub-crate, not a bin inside `rsiprtp`

Putting the binary under `crates/rsiprtp/src/bin/` would pull `chrono`
and any other reporter deps into rsiprtp's published `[dependencies]` —
the runner is a dev tool with nothing to do with library users.
A standalone `tools/full_test` crate keeps `rsiprtp`'s dependency
surface clean and stays out of `cargo publish`.

## Passes (in order)

The runner executes these unconditionally unless the matching `--skip`
flag is passed. Each pass times itself, captures stdout+stderr, and
records pass/fail.

1. **`cargo fmt --check`** — fail if formatting drift, list
   offending files.
2. **`cargo clippy --workspace --all-targets -- -D warnings`** —
   warnings-as-errors, capture warning/error lines for the report.
3. **`cargo test -p rsiprtp --all-targets --no-fail-fast -- --test-threads=1`** —
   every unit test in `src/`, every integration test in `tests/*.rs`,
   plus the example-builds. `-p rsiprtp` skips `gabby` (which needs the
   Vosk native lib). `--test-threads=1` matches the rest of the project
   conventions (some integration tests touch shared sockets).
4. **`cargo doc --no-deps -p rsiprtp`** — make sure rustdoc still
   builds. Cheap, catches broken intra-doc links before publish.
5. **`cargo llvm-cov --workspace --no-cfg-coverage --json`** —
   extract `lines/functions/regions` totals and percentages.

That's it. No special test categories, no benchmark pass (rsiprtp has
no `benches/` today), no audio fidelity flags, no environment-variable
mode-switching.

The baresip / Asterisk integration tests
(`tests/baresip_integration.rs`, `tests/asterisk_integration.rs`)
already self-skip when their external services aren't reachable —
they cost a couple of seconds each to probe and report `ok`. We do not
add a Docker-up step or a `--with-baresip` flag; the user runs
`docker compose up` separately when they want those tests to do real
work, exactly as `CLAUDE.md` already documents.

## CLI

```
cargo run --release -p full_test [OPTIONS]

OPTIONS:
    --quick             Skip coverage (fastest path: ~2 min vs ~6 min)
    --skip-coverage     Skip cargo llvm-cov
    --skip-quality      Skip cargo fmt + cargo clippy
    --help, -h          Show this help
```

Four flags, all `--skip-*` plus `--quick`. No environment-variable
overrides, no feature flags, no toggles for individual passes beyond
the obvious quality/coverage switches. If someone wants to run a
narrower slice they still have plain `cargo test`.

## Output

```
crates/rsiprtp/tests/results/
  2026.05.02 - full test report.html
  2026.05.02 - full test report (2).html        # second run same day
  ...
```

The file path is computed with `chrono::Local::now()` and incremented
on collision (same logic as `build_report_path` in mddosem). HTML is
self-contained: embedded CSS, no external assets, light/dark theme via
`prefers-color-scheme`. Markdown variant is generated and printed to
stdout for terminal scanning, but only the HTML is persisted — same as
mddosem.

The runner also `println!`s the report path at the end so the user can
click straight to it.

## Report structure

Reusing mddosem's HTML template essentially verbatim, minus the
mddosem-specific sections. The report contains:

1. **Header.** Title, timestamp, git short-hash + branch, run mode
   string (e.g. `Default | Coverage Skipped`).
2. **Dashboard cards.** Three large numbers — Passed / Failed / Skipped
   — across all passes.
3. **Summary table.** One row per pass (Quality, Tests, Doc, Coverage),
   with totals + duration. Failed cells get the red status class.
4. **Code Quality section.** Per-check status + duration. Expanded
   issue lists (truncated at 20 lines) below each failing check.
5. **Test Breakdown by category.** Tests grouped by the first one or
   two `::` segments of the test name. Categories sorted with failing
   ones first, then alphabetical.
6. **Failed Tests.** Flat list of failing test names, each as an
   inline `<code>` block, for quick triage.
7. **Coverage.** Lines / Functions / Regions table with covered / total
   / percentage. Skipped if `cargo llvm-cov` isn't installed (we just
   print a warning, don't fail).
8. **Doc build.** Pass/fail + truncated stderr if it failed.

Sections render in this fixed order, but each is conditionally emitted —
if a pass was skipped, its section is gone, not blank.

## Test categorization

A small `category(test_name) -> &'static str` function with the
crate's actual module shape:

| First path segment | Category    |
|--------------------|-------------|
| `sip::`            | SIP         |
| `transport::`      | Transport   |
| `transaction::`    | Transaction |
| `dialog::`         | Dialog      |
| `session::`        | Session     |
| `sdp::`            | SDP         |
| `rtp::`            | RTP         |
| `srtp::`           | SRTP        |
| `ice::`            | ICE         |
| `media::`          | Media       |
| `core::`           | Core        |
| (else)             | Integration |

Integration tests in `tests/*.rs` come through libtest with no `::`
prefix (e.g. `concurrency_safety::test_…`), so they all bucket under
"Integration". That's fine — the per-test list is still rendered, the
category total just tells you "integration suite has N failures".

This is a lookup, not pattern matching against a list of regexes.
`mddosem`'s `high_level_category` grew to ~25 hand-tuned cases over
time; rsiprtp's module layout is shallow and stable, so we won't.

## Failure handling

- Individual cargo test failures → counted, listed in "Failed Tests".
- Cargo process failure (compile error, segfault) with no individual
  test result → synthesize one `cargo::<phase>::process_failure` test
  marked failed, attach the last 50 lines of stderr (mddosem already
  does this; we copy `record_non_test_cargo_failure` essentially as is).
- `cargo llvm-cov` missing → warn, omit the coverage section, don't
  fail the run.
- `cargo doc` failing → record as a fail, don't abort other passes.
- Exit code: `0` if every recorded pass passed, `1` otherwise.

## Scope decisions (what we are *not* doing)

- **No screenshot capture.** mddosem dumps PNGs from emulator runs
  into the report; rsiprtp tests produce no images.
- **No benchmark pass.** No `benches/` exists today. When one is
  added, this will be a one-screen extension — a `run_benchmarks`
  function and a section in the HTML — not a redesign.
- **No DOSBox-style external reference benchmark.** N/A.
- **No mode matrix.** No interpreter/JIT, no audio diag, no fidelity
  lane. One run = one configuration.
- **No environment-variable mode switching.** mddosem sets
  `MDDOSEM_NO_JIT`, `MDDOSEM_NO_AUDIO`, etc., per pass. rsiprtp tests
  don't gate on env vars, so we don't either.
- **No auto-spinning of Docker / baresip / Asterisk.** That's a
  pre-flight the user does manually, and the tests already self-skip
  cleanly when those services aren't running.
- **No `verify_ordinals` / `litmus` equivalent.** Those are
  domain-specific to mddosem's Win16/DOS conformance work; nothing
  analogous in rsiprtp.
- **No CI integration in this change.** A future GitHub Actions job
  can `cargo run --release -p full_test --` and upload the HTML as an
  artifact, but that's a separate change.
- **No public-api / cargo-deny pass yet.** They're tracked in the
  publishing-prep plan
  (`2026.04.30 - Plan - Publishing prep remaining items.md`); slot
  them in when they actually exist.

## Deliverables

1. **`tools/full_test/Cargo.toml`** — new workspace member, deps:
   `chrono` (for date-stamped paths and report timestamp). No other
   third-party deps needed; html escaping and json scraping are 30
   lines each, copied verbatim from mddosem.
2. **`tools/full_test/src/main.rs`** — single file, ~600 LoC, broken
   into the obvious functions: `parse_options`, `run_fmt_check`,
   `run_clippy`, `run_cargo_test`, `run_doc`, `run_coverage`,
   `parse_test_output`, `categorize_tests`, `generate_html_report`,
   `build_report_path`, `write_report`, `main`.
3. **Workspace `Cargo.toml`** — add `"tools/full_test"` to `members`.
   `default-members` unchanged.
4. **`crates/rsiprtp/tests/results/.gitkeep`** — create the directory
   so the runner has somewhere to write on a fresh checkout. Add the
   directory to `.gitignore` for the `*.html` files (we don't want
   reports in version control) but keep `.gitkeep`.
5. **`README.md`** — one short subsection under a "Testing" heading
   pointing at `cargo run --release -p full_test`. Match the existing
   "Build" / "Run tests" patterns already in the README.
6. **`CLAUDE.md`** — one-line add under Build Commands documenting the
   command.

## Validation

- `cargo run --release -p full_test --skip-coverage --skip-quality`
  on a clean checkout produces an HTML file at the documented path
  whose dashboard matches `cargo test -p rsiprtp` numbers.
- Force a fmt drift (insert `let x=1;` somewhere) → run with quality
  on → report shows `cargo fmt: FAIL (1 issues)` and the offending file.
- Force a test failure → report's Failed Tests section lists it,
  exit code is 1.
- Run twice on the same day → second report lands at
  `... full test report (2).html`, neither overwrites the other.

## Open questions

- **Does `cargo llvm-cov` work uniformly on the team's machines?**
  CLAUDE.md says yes (project-level convention). If not, the runner
  warns and continues, so the worst case is "coverage section missing".
- **Where should the report path live — `crates/rsiprtp/tests/results/`
  (per-crate, mirrors mddosem) or repo-root `tests/results/`?** Going
  with per-crate to match mddosem exactly. The runner uses an explicit
  workspace-relative path computed from `CARGO_MANIFEST_DIR` of the
  `full_test` crate plus `../../crates/rsiprtp/tests/results`.
- **Should we also write a `.md` copy of the report?** mddosem doesn't,
  and the markdown is already echoed to stdout. Skipping it.
