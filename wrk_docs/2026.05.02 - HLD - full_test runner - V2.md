# 2026.05.02 - HLD - full_test runner - V2

Supersedes `2026.05.02 - HLD - full_test runner.md`. The shape stays the
same; this revision corrects the cargo command set against the project's
actual CI conventions, addresses the Windows self-rebuild lock, and
folds in `cargo deny` / `cargo audit` (which V1 missed).

## Goal

A single command that runs the whole rsiprtp test bar — formatter,
clippy, unit + integration tests, doc build, coverage, and the supply-
chain checks (`cargo deny`, `cargo audit`) — and writes a self-
contained, dated HTML report under `crates/rsiprtp/tests/results/`.

Modeled on `mddosem`'s `full_test` (`d:\language\mddosem\src\bin\full_test.rs`).
Same idea, much smaller surface: rsiprtp has none of mddosem's modal
complexity (interpreter vs JIT, ordinal/litmus checks, screenshot
capture, DOOM benchmark), so neither does the runner.

## Shape of the change

A new workspace member `tools/full_test/` containing one `main.rs`. It
shells out to `cargo` with `Command`, parses libtest output, writes
one HTML report, and exits non-zero if anything failed.

```
tools/
  full_test/
    Cargo.toml
    src/
      main.rs
```

`Cargo.toml` workspace `members` gains `tools/full_test`.
`default-members` stays `["crates/rsiprtp"]` so `cargo build` is
unchanged for everyone else; the runner is invoked explicitly with
`cargo run --release -p full_test`.

Reports land at
`crates/rsiprtp/tests/results/YYYY.MM.DD - full test report.html`,
mirroring the mddosem layout (date-stamped, `(N)` suffix on collision,
HTML only).

## Why a separate sub-crate, not a bin inside `rsiprtp`

Putting the binary under `crates/rsiprtp/src/bin/` would pull `chrono`
into rsiprtp's published `[dependencies]` — the runner is a dev tool
with nothing to do with library users. A standalone `tools/full_test`
crate keeps `rsiprtp`'s dependency surface clean and stays out of
`cargo publish`.

## Cargo command set

The runner shells out with the project's *actual* CI idiom, not the
shorter `-p rsiprtp` form. This matters because new workspace members
should be picked up automatically without editing the runner.

Every cargo invocation excludes both `gabby` (Vosk native lib) and
`full_test` itself (see "Self-rebuild lock" below):

```
EXCL="--workspace --exclude gabby --exclude full_test"
```

Pass-by-pass:

| # | Pass         | Command                                                                              |
|---|--------------|--------------------------------------------------------------------------------------|
| 1 | fmt          | `cargo fmt --check`                                                                  |
| 2 | clippy       | `cargo clippy $EXCL --all-targets -- -D warnings`                                    |
| 3 | cargo-deny   | `cargo deny check` *(skip if not installed)*                                         |
| 4 | cargo-audit  | `cargo audit` *(skip if not installed)*                                              |
| 5 | tests        | `cargo test $EXCL --no-fail-fast -- --test-threads=1`                                |
| 6 | doc          | `RUSTDOCFLAGS="-D warnings" cargo doc $EXCL --no-deps`                               |
| 7 | coverage     | `cargo llvm-cov $EXCL [--branch] --json --no-cfg-coverage` *(skip if not installed)* |

Notes:

- **`cargo fmt --check`** matches CI verbatim. It formats the default
  members only — gabby's source is *not* fmt-gated by CI today, so the
  runner doesn't tighten that either.
- **`-- --test-threads=1`** is required because of a known parallelism
  flake in `turn::tests::test_turn_client_new_forced_local_addr_error`
  (documented at `.github/workflows/ci.yml:53`). Same reason, same
  flag.
- **`--no-fail-fast`** so one failing test doesn't mask the rest.
- **`RUSTDOCFLAGS=-D warnings`** is critical — without it `cargo doc`
  silently passes when intra-doc links break. CI sets it; the runner
  must too.
- **`--branch`** is added to `cargo llvm-cov` only when `rustc -V`
  contains "nightly" (mirrors mddosem).
- **Tool detection.** Each of `cargo deny`, `cargo audit`, and
  `cargo llvm-cov` is probed with `cargo <tool> --version`. Missing →
  warn, skip the section, do not fail the run. Same pattern mddosem
  uses for `llvm-cov`.

The baresip integration tests
(`tests/baresip_integration/scenarios/*`) are all marked
`#[ignore = "requires baresip..."]`. `cargo test` skips them by default,
so the runner doesn't need to do anything special. If a user wants to
run them, they invoke `cargo test ... -- --include-ignored` directly,
exactly as today. The Asterisk integration test
(`tests/asterisk_integration.rs`) is not ignored but self-skips with an
early return when `127.0.0.1:5060` doesn't answer an `OPTIONS`.

## Self-rebuild lock (Windows)

The runner is itself a workspace member. When it shells out to
`cargo test --workspace`, cargo wants to recompile *every* member —
including `full_test` itself. On Windows, the running `full_test.exe`
holds a write lock on its own image; the rebuild fails with permission
denied.

Two reasonable fixes; we pick (a):

(a) **`--exclude full_test` everywhere.** Honest, explicit, works on
    every platform. Costs us the ability to write tests inside the
    runner crate, but the runner is small enough that we can keep
    its unit tests in a sibling helper module if we ever want them.

(b) `--lib --tests --examples` (mddosem's approach). Skips bin
    targets implicitly. Less explicit, easier to forget on a future
    pass that legitimately wants `--bins`.

Going with (a). Same exclusion list shows up in every cargo command
in the table above.

## CLI

```
cargo run --release -p full_test [OPTIONS]

OPTIONS:
    --skip-quality      Skip cargo fmt + cargo clippy
    --skip-supply       Skip cargo deny + cargo audit
    --skip-coverage     Skip cargo llvm-cov (the dominant cost: ~3-4 min)
    --help, -h          Show this help
```

Four flags. No `--quick` (mddosem's `--quick` skips ignored tests, and
rsiprtp's main suite has none). No env-var overrides. No feature flags.
If a user wants a narrower slice, plain `cargo test ...` is right
there. Coarse skip-groups are deliberate — splitting "skip-deny" and
"skip-audit" individually would buy nothing since both are sub-second.

## Output

```
crates/rsiprtp/tests/results/
  2026.05.02 - full test report.html
  2026.05.02 - full test report (2).html        # second run same day
  ...
```

Path computed from `chrono::Local::now()`, incremented on collision,
same logic as `build_report_path` in mddosem. HTML is self-contained:
embedded CSS, no external assets, light/dark via `prefers-color-scheme`.
A markdown variant is generated and printed to stdout for terminal
scanning, but only the HTML is persisted — same as mddosem.

The runner `println!`s the report path at the end so the user can
click straight to it.

## stdout discipline during a run

mddosem's runner pipes child stdout/stderr (no streaming), which means
a 6-minute coverage pass *looks dead* from the outside. We adopt the
same capture model — necessary to write the report — but emit a
heartbeat from the parent before each pass:

```
=== rsiprtp Full Test Suite ===

[1/7] cargo fmt --check ...                       PASS in 1.2s
[2/7] cargo clippy ...                            PASS in 14.7s
[3/7] cargo deny check ...                        SKIP (not installed)
[4/7] cargo audit ...                             PASS in 0.8s
[5/7] cargo test ...                              running...
       PASS in 92.4s (482 / 482)
[6/7] cargo doc --no-deps ...                     PASS in 21.0s
[7/7] cargo llvm-cov ...                          running...
       72.3% lines, 68.1% regions in 218.6s
Report saved to: ...\2026.05.02 - full test report.html
```

Lines are `\r`-truncated where they need to be progress-y; results are
final and printed once each pass exits.

## Report structure

Reusing mddosem's HTML template essentially verbatim, minus the
mddosem-specific sections. The report contains:

1. **Header.** Title, timestamp, git short-hash + branch, run mode
   string (e.g. `Default | Coverage Skipped`).
2. **Dashboard cards.** Three large numbers — Passed / Failed / Skipped
   — across all passes.
3. **Summary table.** One row per pass (fmt, clippy, deny, audit,
   tests, doc, coverage). Failed cells get the red status class.
   Skipped passes show `SKIP (reason)`.
4. **Code Quality.** fmt + clippy + deny + audit, status + duration.
   Truncated issue lists (≤20 lines) under each failing check.
5. **Test Breakdown by category.** Tests grouped by the first one or
   two `::` segments of the name. Categories sorted with failing ones
   first, then alphabetical.
6. **Failed Tests.** Flat list of failing names, each as inline
   `<code>`, for quick triage.
7. **Coverage.** Lines / Functions / Regions table with covered /
   total / percentage. Branches row added only when `--branch` was
   used (nightly).
8. **Doc build.** Pass/fail + truncated stderr if it failed.

Sections render in this fixed order, but each is conditionally emitted
— if a pass was skipped, its section is gone, not blank.

## Test categorization

A small `category(test_name) -> &'static str` lookup keyed off the
crate's actual module shape:

| First path segment | Category    |
|--------------------|-------------|
| `sip::`            | SIP         |
| `transport::`      | Transport   |
| `transaction::`    | Transaction |
| `dialog::`         | Dialog      |
| `session::`        | Session     |
| `sdp::`            | SDP         |
| `rtp::`            | RTP         |
| `srtp::`           | SRTP        |
| `ice::`            | ICE         |
| `media::`          | Media       |
| `core::`           | Core        |
| (else)             | Integration |

Integration tests in `tests/*.rs` come through libtest with no `::`
prefix (e.g. `concurrency_safety::test_…`), so they all bucket under
"Integration". Per-test list still renders; the category total just
tells you "integration suite has N failures". Lookup, not regex —
mddosem's `high_level_category` grew to ~25 hand-tuned cases over
time; rsiprtp's module layout is shallow and stable.

## Failure handling

- Individual cargo test failures → counted, listed in "Failed Tests".
- Cargo process failure (compile error, segfault) with no individual
  test result → synthesize one `cargo::<phase>::process_failure` test
  marked failed, attach the last 50 lines of stderr (copy
  `record_non_test_cargo_failure` from mddosem essentially as is).
- Missing optional tools (`cargo-deny`, `cargo-audit`, `cargo-llvm-cov`)
  → warn, omit the section, do not fail the run.
- `cargo doc` failing → record as a fail, don't abort other passes.
- Exit code: `0` if every recorded pass passed, `1` otherwise.

## Scope decisions (what we are *not* doing)

- **No `--quick`**. mddosem's `--quick` skips ignored tests; rsiprtp's
  main suite has none. The flag would do nothing.
- **No screenshot capture.** rsiprtp tests produce no images.
- **No benchmark pass.** No `benches/` exists today. When one is
  added, this is a one-screen extension — a `run_benchmarks` function
  and a section in the HTML — not a redesign.
- **No mode matrix.** No interpreter/JIT, no audio diag, no fidelity
  lane. One run = one configuration.
- **No environment-variable mode switching.** rsiprtp tests don't gate
  on env vars, so we don't either.
- **No auto-spinning of Docker / baresip / Asterisk.** Pre-flight is
  the user's responsibility; the tests gracefully skip otherwise.
- **No `verify_ordinals` / `litmus` equivalent.** Domain-specific to
  mddosem.
- **No fuzz pass.** `fuzz/` has 23 targets (`rtp_packet`, `sip_message`,
  …); they are designed to run for hours-to-overnight, not as part of
  a "run all tests" cycle. Fuzzing keeps its existing manual cadence.
- **No MSRV check.** CI runs `cargo build --locked` against Rust 1.88
  via `dtolnay/rust-toolchain@1.88.0`. Reproducing that locally needs
  toolchain juggling that the runner shouldn't try to do silently. Out.
- **No `cargo public-api` baseline.** Tracked in
  `2026.04.30 - Plan - Publishing prep remaining items.md`; slot in
  when it actually exists.
- **No CI integration in this change.** A future GitHub Actions job
  can `cargo run --release -p full_test --` and upload the HTML as an
  artifact. Separate change.

## Deliverables

1. **`tools/full_test/Cargo.toml`** — new workspace member, deps:
   `chrono` only. HTML escaping and JSON scraping are ~30 lines each,
   copied verbatim from mddosem; no third-party need.
2. **`tools/full_test/src/main.rs`** — single file, ~600 LoC, broken
   into the obvious functions: `parse_options`, `run_fmt_check`,
   `run_clippy`, `run_cargo_deny`, `run_cargo_audit`, `run_cargo_test`,
   `run_doc`, `run_coverage`, `parse_test_output`, `categorize_tests`,
   `generate_html_report`, `build_report_path`, `write_report`, `main`.
3. **Workspace `Cargo.toml`** — add `"tools/full_test"` to `members`.
   `default-members` unchanged.
4. **`crates/rsiprtp/tests/results/.gitkeep`** — create the directory
   so the runner has somewhere to write on a fresh checkout.
5. **`.gitignore`** — add `crates/rsiprtp/tests/results/*.html` so
   reports don't end up in commits.
6. **`README.md`** — short subsection under "Testing" pointing at
   `cargo run --release -p full_test`, matching the existing build /
   test patterns already in the README.
7. **`CLAUDE.md`** — one-line add under Build Commands documenting the
   command.

## Validation

- `cargo run --release -p full_test --skip-coverage --skip-quality
  --skip-supply` on a clean checkout produces an HTML file at the
  documented path whose totals match `cargo test --workspace --exclude
  gabby --exclude full_test -- --test-threads=1`.
- Force a fmt drift (insert `let x=1;` somewhere) → run with quality
  on → report shows `cargo fmt: FAIL (1 issues)` and the offending file;
  exit code 1.
- Force a test failure → report's Failed Tests section lists it; exit
  code 1.
- Run twice on the same day → second report lands at
  `... full test report (2).html`, neither overwrites the other.
- On a Windows machine, run while previous `full_test.exe` is still
  loaded as a process → no permission-denied errors during cargo test
  (the `--exclude full_test` exclusion makes this a non-event).
- Uninstall `cargo-deny` → run → deny pass shows `SKIP (not installed)`,
  exit code unchanged.

## Open questions

- **Where should the report path live — `crates/rsiprtp/tests/results/`
  (per-crate, mirrors mddosem) or repo-root `tests/results/`?** Going
  with per-crate to mirror mddosem exactly. The runner uses an explicit
  workspace-relative path computed from `CARGO_MANIFEST_DIR` of the
  `full_test` crate plus `../../crates/rsiprtp/tests/results`.
- **Should we also write a `.md` copy?** mddosem doesn't, and the
  markdown is already echoed to stdout. Skipping it.
- **Should `cargo deny` failures be hard-fail or warn?** CI hard-fails.
  The runner does too — surfacing license/advisory drift early is the
  whole point of having `deny.toml`. (`RUSTSEC-2023-0071` and
  `RUSTSEC-2025-0134` are already ignored in `deny.toml`, so steady
  state is green.)
