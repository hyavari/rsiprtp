# Briefing: Wire CongestionController target bitrate into the Opus encoder

## Why this matters

REMB parsing (RFC 4585 + draft-alvestrand-rmcat-remb) and an AIMD `CongestionController` already exist in `mdsiprtp-rtp`, but nothing applies the produced `target_bitrate` to any encoder. Without this bridge, the REMB plumbing is dead code ŌĆö the whole point of receiver-estimated maximum bitrate is closing that loop into the encoder.

A separate task is landing now that swaps the codec backend to `ropus 0.12` and adds runtime setters/getters on `OpusCodec` (`set_bitrate(Bitrate)`, `bitrate()`, `effective_bitrate_bps()`, etc.). Everything below assumes that swap is in. See the HLD next to this file.

## Current state (anchors)

- `crates/mdsiprtp-rtp/src/rtcp.rs:923` ŌĆö `Remb` parse/build.
- `crates/mdsiprtp-rtp/src/session.rs:81` ŌĆö `CongestionController` (AIMD: SlowStart / CongestionAvoidance / Recovery). Methods of interest: `on_remb(&mut self, bitrate: u64)`, `target_bitrate(&self) -> u64`, `state(&self) -> CongestionState`.
- `crates/mdsiprtp-media/src/codec/opus.rs` ŌĆö after the ropus 0.12 swap: `OpusCodec::set_bitrate(&mut self, Bitrate) -> Result<(), String>`, plus matching getter. `Bitrate` is re-exported from `mdsiprtp-media`.
- No trait unifies `G711Codec`, `G722Codec`, `OpusCodec` ŌĆö they are concrete types living in `mdsiprtp-media::codec`.

## Goal

Drive `OpusCodec` bitrate from `CongestionController::target_bitrate()` updates so that REMB-driven adaptation actually changes the encoder rate in flight. Fixed-rate codecs (G.711, G.722) must remain unaffected ŌĆö pushing a target rate at them should be a non-event, not a no-op method on a misleading common trait.

## Design choices to weigh

You will be introducing the first trait that crosses the codec types. Pick the smallest pattern that works.

1. **`AdaptiveBitrate` trait, implemented only by adaptive codecs** (recommended starting point). Caller holds an `Option<&mut dyn AdaptiveBitrate>` resolved at codec selection time, or matches on the concrete codec at the session layer. Honest about which codecs adapt; no no-op methods on G.711 / G.722.
2. **Unified `Codec` trait with a `set_target_bitrate` no-op for fixed-rate codecs.** Easier to call generically, but hides whether adaptation worked. Misleads readers into thinking they can adapt G.711.
3. **Subscription channel.** `CongestionController` exposes a `watch::Receiver<u64>`; codec subscribes on construction. Decouples cleanly but adds plumbing and async-vs-sync coupling that the rest of the codec layer does not have today.

#1 is almost certainly right. Confirm by reading how the session layer currently holds the encoder and the CC together, then pick.

## Open design questions to answer before coding

- **Where does the wiring live?** Most likely the session layer (`mdsiprtp-session`), since that is where call lifecycle, codec selection, and the RTP/RTCP loops already meet. Confirm ŌĆö do not sprinkle bridge logic across crates.
- **Update cadence.** Every CC update? Every N frames? Only on threshold crossing? Recommend a small hysteresis (e.g., suppress changes < 5% relative, or rate-limit to once per ~200 ms) so trivial REMB jitter does not churn the encoder.
- **Bandwidth (NB/MB/WB/SWB/FB) reselection at low bitrates.** Out of scope for v1, but `Encoder::set_max_bandwidth` exists and the topic should at least appear in your design notes so a follow-up task knows where to start.
- **Floor / ceiling.** The CC has `min_bitrate` / `max_bitrate`. The Opus encoder has its own valid range. Decide where clamping happens; do it once.

## Testability ŌĆö this is the bar

- **Unit**: bridge correctly maps `u64` bps from CC to `Bitrate::Bits(_)` on the codec, applies updates monotonically, respects hysteresis if added.
- **Unit**: `CongestionController` already has tests for state transitions; do not duplicate.
- **Integration** (the meaningful test): drive a synthetic REMB packet through the RTCP path, observe that the next encoded RTP packet reflects the new target rate. The end-to-end SIP harness lives at `crates/mdsiprtp/tests/baresip_integration/`; you almost certainly want a smaller, codec-focused integration test that does not require Asterisk ŌĆö something that wires `RtpSession` + `CongestionController` + `OpusCodec` directly and feeds it parsed REMB bytes.
- ropus is bit-exact against the xiph reference and has its own RFC 6716 test vectors. Do not re-verify the codec; verify the bridge.

## Constraints

- Pre-publication, dev sandbox: no migration shims, no compatibility flags, no env vars.
- Public API of `mdsiprtp-media` stays clean ŌĆö add a trait if you need one; do not bolt bridge-specific methods onto `OpusCodec`.
- `G711Codec` and `G722Codec` must not gain no-op methods.

## Out of scope

- Generating outbound REMB (we only consume incoming).
- Transport-wide congestion control (RFC 8888 / TCC).
- Bandwidth or payload-type renegotiation via re-INVITE.
- Adapting G.722 (no codec-level knob exists; nothing to do).
