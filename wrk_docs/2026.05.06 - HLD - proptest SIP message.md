## 1. Scope

**In scope.** Add a single `cargo test`-driven proptest property test that:

1. Procedurally generates **structurally-valid SIP requests and responses** as `Vec<u8>` (canonical CRLF wire bytes).
2. Hands the bytes to the existing oracle `parser_roundtrip_oracle::assert_roundtrip_fixed_point`.
3. On failure, leverages proptest's shrinker to minimize the offending message.

**Oracle reused, not reinvented.** The Tier-1 fixed-point oracle at `crates/rsiprtp/tests/parser_roundtrip_oracle/mod.rs` is the single assertion. We do **not** add a parallel oracle, do **not** mutate the oracle, and do **not** assert any property the oracle does not already assert.

**Explicit non-goal (Track B's problem):** Tier-2 typed-header round-trip — `Header::typed_from() ∘ From::to_string()` and friends — is **out of scope here**. Track A asserts only Tier-1 (`Message::parse ∘ Message::to_bytes` fixed-point). If a generated `From:` value happens to be Tier-2-illegal, that does not break this track because the oracle never invokes `.typed()`. Track B exercises Tier-2 separately on per-header generators.

**Why this complements libfuzzer.** The existing libfuzzer target `fuzz/fuzz_targets/sip_message_roundtrip.rs` mutates raw bytes; ~99% of its mutants fail at framing and the oracle short-circuits. proptest exercises the *message space*: legal method × legal status × header set × header value family × body shape combinations that bit-mutation on a small seed corpus will not reach in reasonable time. Shrinking on failure produces minimal, copy-pasteable repros — invaluable for triage compared to libfuzzer's crash artifacts (which on Windows-MSVC the M11 work showed don't even get written for `__fastfail` panics).

## 2. Strategy tree

The top-level strategy is `Strategy<Value = Vec<u8>>` whose value is canonical SIP wire bytes. It branches on request-vs-response with equal weight:

```rust
fn arb_message() -> impl Strategy<Value = Vec<u8>> {
    prop_oneof![
        1 => arb_request_bytes(),
        1 => arb_response_bytes(),
    ]
}
```

### 2.1 Method selection

```rust
fn arb_method_token() -> impl Strategy<Value = String> {
    prop_oneof![
        // 90% — recognized methods. The 14 native variants from
        // src/sip/parser/method.rs.
        18 => prop::sample::select(&[
            "INVITE", "ACK", "BYE", "CANCEL", "REGISTER", "OPTIONS",
            "INFO", "UPDATE", "REFER", "NOTIFY", "SUBSCRIBE", "PRACK",
            "MESSAGE", "PUBLISH",
        ]).prop_map(str::to_owned),
        // 10% — case-variant of a recognized method (Method::FromStr is
        // case-insensitive per src/sip/parser/method.rs:97). Tests that
        // round-trip canonicalizes case.
        1 => prop::sample::select(&["invite", "Invite", "iNvItE", "bye", "publish"])
            .prop_map(str::to_owned),
        // Note: there is NO "unknown method" tail. parse_request_line
        // calls Method::from_str which rejects unknown tokens with
        // SipError::Parse, so the oracle would short-circuit on the
        // first parse and the test would degenerate into "we
        // generated bytes the parser silently discarded". Generators
        // that ALL pass first-parse are the high-signal generators.
    ]
}
```

**Decision recorded:** no "unknown method" tail. Reasoning above — they're discarded by oracle's first-parse check. If supervisor wants exploration of the parse-rejection path, that's a different test (a "parser does not panic on garbage" test) which is what the libfuzzer target already does.

### 2.2 Request-URI

The request-URI lives in a `String` field but is *validated* at parse time by `framing::parse_request_line` calling `SipUri::parse`. So the generator must emit URIs `SipUri::parse` accepts. Anything else fails first-parse and the oracle short-circuits — wasted samples.

```rust
fn arb_user() -> impl Strategy<Value = String> {
    // RFC 3261 §25 user characters: unreserved + escaped + user-unreserved.
    // For Track A we keep it simple. SipUri parse-acceptance is the only
    // hard constraint; a fuller alphabet is Track B's URI generator job.
    "[a-zA-Z0-9_.~+-]{1,12}".prop_map(str::to_owned)
}

fn arb_host() -> impl Strategy<Value = String> {
    prop_oneof![
        // hostname-shaped — most common
        4 => "[a-z][a-z0-9-]{1,8}(\\.[a-z][a-z0-9-]{1,8}){0,3}".prop_map(str::to_owned),
        // dotted-quad IPv4 literal
        2 => (0u8..=255, 0u8..=255, 0u8..=255, 0u8..=255)
            .prop_map(|(a,b,c,d)| format!("{a}.{b}.{c}.{d}")),
        // bracketed IPv6 — minimal forms only; full IPv6 is overkill for Track A
        1 => prop::sample::select(&["[::1]", "[2001:db8::1]", "[fe80::1]"])
            .prop_map(str::to_owned),
    ]
}

fn arb_port() -> impl Strategy<Value = Option<u16>> {
    prop::option::weighted(0.3, 1u16..=65535)
}

fn arb_request_uri() -> impl Strategy<Value = String> {
    prop_oneof![
        // 70% — sip:user@host[:port], the dominant real-world shape
        7 => (arb_user(), arb_host(), arb_port())
            .prop_map(|(u,h,p)| match p {
                Some(port) => format!("sip:{u}@{h}:{port}"),
                None       => format!("sip:{u}@{h}"),
            }),
        // 15% — sips:user@host[:port]
        2 => (arb_user(), arb_host(), arb_port())
            .prop_map(|(u,h,p)| match p {
                Some(port) => format!("sips:{u}@{h}:{port}"),
                None       => format!("sips:{u}@{h}"),
            }),
        // 10% — sip:host (no user) — legal per RFC 3261 §19.1
        1 => arb_host().prop_map(|h| format!("sip:{h}")),
        // 5% — tel: URI (RFC 3966) — accepted by parse_request_line per
        // tests/fixtures + framing.rs:test_request_line_accepts_tel_uri
        1 => "(\\+?[1-9][0-9]{6,14})".prop_map(|n| format!("tel:{n}")),
    ]
}
```

**Punted to Track B.** URI parameters (`;transport=tcp;lr`), URI headers (`?Subject=Hi&Body=`), bracketed userpart-with-`;`, IPv6 zone-id, multi-segment user-info. The `SipUri` parser handles all of these but exercising them here doubles generator complexity for marginal Tier-1 signal. Track B owns `SipUri` round-trip on its own, and Track A's oracle is unaffected by what the URI string *contains* as long as it survives `SipUri::parse`.

### 2.3 Status code & reason phrase (responses)

```rust
fn arb_status_code() -> impl Strategy<Value = u16> {
    prop_oneof![
        // 80% — codes the parser knows (StatusCode constants) plus
        // any value in [100, 699]. The parser accepts ANY u16 in
        // 100..=699 per framing.rs:245. Use the full legal range to
        // exercise the unknown-code path (StatusCode::new + "Unknown"
        // reason phrase).
        4 => 100u16..=699,
        // 20% — concentrate on the well-known codes that ship as
        // associated constants. These are the realistic-traffic shapes.
        1 => prop::sample::select(&[
            100u16, 180, 183, 200, 202, 301, 302, 401, 403, 404,
            407, 408, 480, 486, 487, 488, 500, 503, 504, 603,
        ]),
    ]
}

fn arb_reason_phrase() -> impl Strategy<Value = String> {
    // RFC 3261 §25.1 grammar for Reason-Phrase: reserved / unreserved /
    // escaped / UTF8-NONASCII / UTF8-CONT / SP / HTAB.
    // CR, LF, NUL, DEL, all CTL bytes other than HTAB are FORBIDDEN
    // (parse_status_line rejects them — see framing.rs:256-265, pinned
    // by tests test_parse_status_line_reason_with_bare_*_rejects).
    // We must NOT generate forbidden bytes — the oracle would short-
    // circuit on first parse. Restrict to printable ASCII + space + tab.
    prop_oneof![
        4 => "[A-Za-z0-9 ]{0,32}".prop_map(str::to_owned),
        1 => Just("".to_string()),                   // empty reason — RFC-legal
        1 => Just("Busy Here".to_string()),
        1 => Just("OK".to_string()),
        1 => "[A-Za-z0-9\t ]{0,32}".prop_map(str::to_owned),  // tab-bearing
    ]
}
```

**Note on reason normalization.** The serializer emits `self.reason` verbatim — it does NOT canonicalize via `StatusCode::reason_phrase()`. So a generated `(code=200, reason="Busy")` will round-trip with reason "Busy", not "OK". Fixed point holds at m1.

### 2.4 Headers — the central design choice

We must generate header *values* without re-implementing the typed parsers. Two channels:

**Channel A — native variant with curated literal value.** For each of the 19 native variants in `Header`, supply a small set (3–5 items) of values known to round-trip cleanly. These come from the existing fixture corpus (`tests/fixtures/handcrafted/*.sip` and the unit-test bodies in `header.rs`/`message.rs`).

```rust
fn arb_native_header() -> impl Strategy<Value = Header> {
    use rsiprtp::sip::parser::Header;

    let allow      = prop::sample::select(&["INVITE, ACK", "INVITE, ACK, BYE, CANCEL", "INVITE"]);
    let auth       = prop::sample::select(&[
        "Digest realm=\"x\", nonce=\"abc\"",
        "Digest username=\"a\", realm=\"x\", nonce=\"y\", uri=\"sip:a@b\", response=\"deadbeef\"",
    ]);
    let cseq       = (1u32..=999_999, prop::sample::select(&[
        "INVITE","ACK","BYE","CANCEL","REGISTER","OPTIONS","INFO","UPDATE","REFER",
        "NOTIFY","SUBSCRIBE","PRACK","MESSAGE","PUBLISH"
    ])).prop_map(|(n,m)| format!("{n} {m}"));
    let call_id    = "[a-zA-Z0-9]{8,16}@[a-z][a-z0-9.-]{2,12}".prop_map(str::to_owned);
    let contact    = (arb_user(), arb_host()).prop_map(|(u,h)| format!("<sip:{u}@{h}>"));
    // Content-Length intentionally omitted from this set — the message-
    // builder synthesizes it from the actual body length so wire bytes
    // are always consistent. See §2.5.
    let content_type = prop::sample::select(&[
        "application/sdp", "text/plain", "application/dtmf-relay",
        "message/sipfrag", "application/pidf+xml",
    ]);
    let expires    = (0u32..=86_400).prop_map(|n| n.to_string());
    let from_to    = (arb_user(), arb_host(), "[a-f0-9]{6,12}")
        .prop_map(|(u,h,tag)| format!("<sip:{u}@{h}>;tag={tag}"));
    let max_fwd    = (0u8..=70).prop_map(|n| n.to_string());
    let route      = (arb_user(), arb_host()).prop_map(|(u,h)| format!("<sip:{u}@{h};lr>"));
    let require    = prop::sample::select(&["100rel", "timer", "path", "100rel, timer"]);
    let supported  = prop::sample::select(&["timer", "100rel", "path", "replaces", ""]);
    let via        = (arb_host(), "[a-zA-Z0-9]{16,24}")
        .prop_map(|(h,b)| format!("SIP/2.0/UDP {h};branch=z9hG4bK{b}"));
    let www_auth   = prop::sample::select(&["Digest realm=\"x\", nonce=\"y\""]);

    prop_oneof![
        2 => allow.prop_map(|s| Header::Allow(s.into())),
        2 => auth.prop_map(|s| Header::Authorization(s.into())),
        4 => cseq.prop_map(Header::CSeq),
        4 => call_id.prop_map(Header::CallId),
        4 => contact.prop_map(Header::Contact),
        // ContentLength: skipped — see §2.5.
        3 => content_type.prop_map(|s| Header::ContentType(s.into())),
        2 => expires.prop_map(Header::Expires),
        4 => from_to.clone().prop_map(Header::From),
        4 => from_to.prop_map(Header::To),
        3 => max_fwd.prop_map(Header::MaxForwards),
        2 => prop::sample::select(&["Digest"]).prop_map(|s| Header::ProxyAuthenticate(s.into())),
        2 => prop::sample::select(&["Digest"]).prop_map(|s| Header::ProxyAuthorization(s.into())),
        2 => route.clone().prop_map(Header::RecordRoute),
        2 => require.prop_map(|s| Header::Require(s.into())),
        2 => route.prop_map(Header::Route),
        2 => supported.prop_map(|s| Header::Supported(s.into())),
        4 => via.prop_map(Header::Via),
        2 => www_auth.prop_map(|s| Header::WwwAuthenticate(s.into())),
    ]
}
```

**Channel B — `Other(name, value)` for unknown headers.** Header names that don't resolve to one of the 19 native variants land in `Header::Other`. These ride through Tier-1 entirely as opaque strings, so any printable-ASCII value round-trips trivially.

```rust
fn arb_other_header() -> impl Strategy<Value = Header> {
    let name = "[A-Z][A-Za-z]{1,8}(-[A-Z][A-Za-z]{1,8}){0,2}";
    // Avoid generating names that resolve to one of the 19 native
    // variants — that would re-route via the native channel during
    // round-trip and the canonicalization (Other -> typed) would
    // surface a different shape on m2 vs m1. Use an X- prefix or
    // names known to be outside the native set.
    let safe_name = prop_oneof![
        4 => prop::sample::select(&[
            "User-Agent", "Server", "Subject", "Date", "Organization",
            "Priority", "Reply-To", "Timestamp",
        ]).prop_map(str::to_owned),
        3 => "X-[A-Z][A-Za-z]{1,8}".prop_map(str::to_owned),
    ];
    let value = "[ -~]{0,40}".prop_filter("no leading/trailing whitespace, no colon edge",
        |s: &String| !s.starts_with(' ') && !s.ends_with(' '));
    (safe_name, value).prop_map(|(n,v)| Header::Other(n, v))
}
```

**Important caveat on the value alphabet.** The header parser is *lenient*: it accepts NUL bytes (pinned in `framing.rs:test_header_with_embedded_nul_pinned_accepted`), embedded CR fragments, etc. We **must not** generate values containing CR or LF — those bytes are line terminators in the wire form and will reframe the message on round-trip. Specifically:

- LF: `parse_header_block` calls `str::lines` which splits on LF. A header with a literal LF in its value will be split into two lines; the second line will likely fail `parse_line` (no colon) or be absorbed as a folded continuation if the first byte is space/tab. Either way the post-round-trip AST diverges.
- CR: similarly absorbed by `\r\n` recognition.

Solution: restrict header-value alphabet to `[\x20-\x7E\t]` (printable ASCII + tab). NUL and high-bit UTF-8 are technically accepted by the parser but high-bit UTF-8 must be syntactically valid (the framing layer requires the header section to be valid UTF-8 — `framing.rs:39-40`). Track A skips both. Track B can claim them if it wants extra adversarial mileage on the typed parsers.

**Header-set strategy.**

```rust
fn arb_header_set() -> impl Strategy<Value = Vec<Header>> {
    // Realistic sets are 5-12 headers. The MAX_HEADERS cap is 256;
    // we stay well below that for runtime speed. Required headers
    // for the message to be "interesting" but not over-constrained:
    // for now we don't enforce From/To/Call-ID/CSeq presence (the
    // Tier-1 parser doesn't require them). Track A intentionally
    // does NOT enforce request-message semantic minima — that's a
    // higher-level oracle's concern.
    prop::collection::vec(
        prop_oneof![
            7 => arb_native_header(),
            3 => arb_other_header(),
        ],
        0..=12,
    )
}
```

### 2.5 Body & Content-Length

Two design choices here, and we lean toward the **second** (always emit consistent Content-Length):

| Choice | Pros | Cons |
|--------|------|------|
| Lean on the parser's stale-CL renormalization | Exercises the renormalization codepath — Tier-1's only legitimate first-round-trip mutation that affects bytes | Forces every test sample to tolerate m1 ≠ m2 (which the oracle does — fine) but hides the simpler property that consistent inputs are a true fixed point at m1 |
| **Always emit Content-Length consistent with body** (chosen) | Simpler invariant: m1 == m2 == m3 expected for ~95% of samples. Stale-CL renormalization is already covered by the static fixture `rt_oracle_holds_after_normalization`. | Doesn't probe stale-CL via proptest. Fine — the static fixture pins it and libfuzzer mutates it. |

Implementation:

```rust
fn arb_body() -> impl Strategy<Value = Vec<u8>> {
    prop_oneof![
        // 50% — empty body (the dominant case for non-INVITE traffic)
        4 => Just(Vec::new()),
        // 30% — small text body, printable ASCII, no CRLF
        3 => "[ -~]{1,200}".prop_map(|s| s.into_bytes()),
        // 15% — small octet-string body (any bytes, including high-bit
        // and NUL — bodies bypass UTF-8 validation per framing.rs:38-39)
        2 => prop::collection::vec(any::<u8>(), 1..=200),
        // 5% — minimal SDP-shaped body. Cheap literal; Track C owns
        // SDP generation. This is just one literal so we exercise the
        // application/sdp Content-Type pairing without coupling tracks.
        1 => Just(b"v=0\r\no=- 1 1 IN IP4 127.0.0.1\r\ns=-\r\nc=IN IP4 127.0.0.1\r\nt=0 0\r\n".to_vec()),
    ]
}
```

After the headers and body are decided, the assembler **always inserts a fresh `Content-Length` header** equal to `body.len()`. If the random header set happened to include a curated Content-Length (it doesn't — we omit it from `arb_native_header`), we'd dedupe. By construction we won't.

### 2.6 Line endings

**On the wire we generate canonical `\r\n` only.** Mixed `\n` line endings are accepted by the framer (`framing.rs:60-66`, `split_lines` uses `str::lines` which tolerates both) but their round-trip semantics are fully covered by static fixtures. Generating only canonical bytes keeps the property simple: m1 should be a true fixed point against m1.to_bytes() in the consistent-CL case. If supervisor wants to crank coverage on the `\n`-tolerance path that's a 5-line generator extension, deferred.

### 2.7 Assembly

```rust
fn assemble_request(method: String, uri: String, headers: Vec<Header>, body: Vec<u8>) -> Vec<u8> {
    let mut out = Vec::with_capacity(256 + body.len());
    out.extend_from_slice(method.as_bytes());
    out.push(b' ');
    out.extend_from_slice(uri.as_bytes());
    out.push(b' ');
    out.extend_from_slice(b"SIP/2.0\r\n");
    for h in &headers {
        out.extend_from_slice(h.name().as_bytes());
        out.extend_from_slice(b": ");
        out.extend_from_slice(h.value().as_bytes());
        out.extend_from_slice(b"\r\n");
    }
    out.extend_from_slice(b"Content-Length: ");
    out.extend_from_slice(body.len().to_string().as_bytes());
    out.extend_from_slice(b"\r\n\r\n");
    out.extend_from_slice(&body);
    out
}

fn arb_request_bytes() -> impl Strategy<Value = Vec<u8>> {
    (arb_method_token(), arb_request_uri(), arb_header_set(), arb_body())
        .prop_map(|(m,u,h,b)| assemble_request(m,u,h,b))
}

fn arb_response_bytes() -> impl Strategy<Value = Vec<u8>> {
    (arb_status_code(), arb_reason_phrase(), arb_header_set(), arb_body())
        .prop_map(|(c,r,h,b)| assemble_response(c,r,h,b))
}
```

Total generator surface: ~150 lines of helper code.

## 3. Oracle

```rust
oracle::assert_roundtrip_fixed_point(&bytes);
```

That is the *entire* assertion. We do NOT add comparisons against the input bytes (the oracle's whole point is that m1 may differ from m2; asserting input == m1 would be wrong). We do NOT add separate `assert!`s on parse success; the oracle handles it (silently returning on first-parse failure is the design, but in this track we expect first-parse to succeed by construction — see §6 on coverage stats).

If the supervisor wants telemetry on "what fraction of generated samples actually reach the m2 assertion vs short-circuit on first parse", that's added with a `prop_assume!(Message::parse(&bytes).is_ok())` *before* the oracle call so proptest tallies discards. Recommended for the first run; remove once we trust the generator.

## 4. Shrinking

Proptest shrinks by walking back along each strategy's shrink chain. The generators above use prop-test-native combinators (`prop_oneof!`, range types, regex strategies via `prop_regex` if we use the `proptest-regex` crate, otherwise hand-rolled), so shrinking is automatic.

**Expected shrunk failure shape.**

A counterexample reduces to:

- **Method:** drops to `INVITE` (first item of the `select` list shrinks toward index 0; `prop::sample::select` shrinks index, so the first entry is the shrink target).
- **Request-URI:** shrinks toward `sip:a@a` or similar — short user, single-letter host. The `prop_oneof!` in `arb_request_uri` shrinks toward branch 0 (the `sip:user@host` shape) and then within that, regex and `arb_host` shrink toward minimal forms.
- **Status code:** shrinks toward 100 (low end of the range).
- **Reason phrase:** shrinks toward `""` (empty regex match).
- **Header set:** shrinks toward `vec![]`. If the failure depends on a particular header, proptest's `prop::collection::vec` shrinker bisects the vec and you end up with the minimal subset that reproduces.
- **Body:** shrinks toward empty.

**Invariants shrinking MUST preserve.** We rely on proptest's structural shrinking — it preserves the *generator structure*, not free-form invariants. So:

- The start line never shrinks to empty (method is from a curated list; URI regex has min-length 1 on user; host regex has min-length 1; the assembler never emits a malformed start line).
- Content-Length always reflects body length (it's recomputed in `assemble_*`, not generated independently).
- Header `name: value\r\n` shape is enforced by the assembler — shrinking a header's value to empty produces `Name: \r\n`, which is legal.
- CRLF blank line separator is unconditional in the assembler.

If shrinking ever produces an input that fails first-parse, the oracle silently returns — meaning the failure shrunk past the bug. To mitigate: when triaging, consult `proptest-regressions/` (the persistent regressions file proptest writes on failure) which contains the *original* failing seed, and rerun without shrinking using `PROPTEST_DISABLE_FAILURE_PERSISTENCE=0 PROPTEST_MAX_SHRINK_ITERS=0`.

**One pre-known shrinking caveat.** If a failure depends on the *combination* of (header value contains some character, status code is N), proptest's component-wise shrinker may shrink them independently. That is the proptest API; no remediation possible at the strategy level. Document in the test docstring.

## 5. File layout

```
crates/rsiprtp/
├── Cargo.toml                    -- ADD `proptest = "1"` to [dev-dependencies]
└── tests/
    ├── parser_roundtrip_oracle/
    │   └── mod.rs                -- UNCHANGED (existing oracle)
    ├── parser_roundtrip.rs       -- UNCHANGED (static driver)
    └── proptest_sip_message.rs   -- NEW (this track)
```

**Single file, no helper module.** The generator helpers (~150 LOC) live inline in `proptest_sip_message.rs`. The HLD originally suggested a `proptest_sip_message_gen/mod.rs` mirror of the oracle layout, but:

- The oracle uses a subdir specifically because it's *shared* between the integration test and a fuzz target via `#[path]`. The generator is not shared — Track B has its own (per-header) generators, Track C has its own (SDP). So no `#[path]` reuse pattern.
- Cargo's integration-test discovery rule (top-level `.rs` files in `tests/`) means a single-file integration test is the simplest layout.
- ~150 LOC inline is well within the readability budget.

If supervisor disagrees, the alternative is `tests/proptest_sip_message_gen/mod.rs` brought in by `proptest_sip_message.rs` via `#[path]`, exactly mirroring the oracle pattern. No functional difference.

**Oracle reuse.** The new test file consumes the oracle the same way `parser_roundtrip.rs` does:

```rust
#[path = "parser_roundtrip_oracle/mod.rs"]
mod oracle;

use oracle::assert_roundtrip_fixed_point;
```

This is **exactly** the existing pattern. We reach the existing oracle module via `#[path]`, do not duplicate it, do not move it.

**Cargo.toml change** — single line in `[dev-dependencies]`:

```toml
proptest = "1"
```

Latest stable is `1.5` as of 2026-05; using `"1"` accepts any 1.x, which is the project convention (`tempfile = "3"`, `chrono = "0.4"`). The crate is pure-Rust, no native deps. Adds ~3-5s to first compile of `cargo test -p rsiprtp` but is cached thereafter.

## 6. Run shape

### 6.1 Default `cargo test` invocation

```rust
proptest! {
    #![proptest_config(ProptestConfig {
        cases: 256,
        max_shrink_iters: 4096,
        max_global_rejects: 65_536,
        .. ProptestConfig::default()
    })]

    #[test]
    fn rt_proptest_sip_message(bytes in arb_message()) {
        oracle::assert_roundtrip_fixed_point(&bytes);
    }
}
```

**Case count: 256.** Proptest's default. Each case parses-serializes-parses-serializes-parses, so per-case work is ~5× a single parse. Estimate per-case wall time on a dev box: ~0.1ms (parse on a ~500-byte message is sub-microsecond per the M11 fuzz numbers — 45k exec/s on libfuzzer = ~22µs per oracle invocation including instrumentation overhead). Total: 256 × 0.1ms ≈ **25ms wall**. Well under the sub-second budget required by `cargo test --workspace`.

**Why 256, not 1024 or 64?** 256 is enough that the prop_oneof! tail branches each fire on average ~26 times (10% × 256), giving each header variant 5–10 firings per `cargo test`. Below 64 the rare branches don't fire reliably; above 1024 we eat into the workspace test budget without compounding signal.

### 6.2 Soak knob

```rust
fn cases() -> u32 {
    std::env::var("RSIPRTP_PROPTEST_CASES")
        .ok()
        .and_then(|s| s.parse().ok())
        .unwrap_or(256)
}
```

Then `ProptestConfig { cases: cases(), .. }`. Nightly soak runs `RSIPRTP_PROPTEST_CASES=1_000_000 cargo test -p rsiprtp --test proptest_sip_message --release`. Estimated wall time: ~100s.

We do NOT set `cases` via proptest's own `PROPTEST_CASES` env var because that affects ALL proptests in the workspace (Tracks B and C will land soon). Per-track env var keeps soak knobs independent.

### 6.3 Workspace test bar

`cargo test --workspace` is part of the standard bar (CLAUDE.md). At 25ms wall this proptest test adds noise-floor overhead. The compile-time cost of the proptest dev-dep is the larger concern; on a clean target dir it adds ~5s. CI caches the target dir so this is a one-time hit per cache-invalidation.

## 7. Anti-scope

Explicitly **NOT** in this track:

- **No SDP body generation.** Bodies are empty, random bytes, or one curated SDP literal. Track C owns SDP. Tying Track A's pass/fail to Track C's correctness creates cross-track flakes.
- **No transport-layer concerns.** No socket tests, no UDP framing, no TCP message-stream framing.
- **No SIPS/TLS specifics.** `sips:` URIs are generated as one branch of `arb_request_uri`, but the test does not exercise certificate validation, the rustls dev-dep, or any TLS handshake. Tier-1 parsing of the URI string is the entire scope.
- **No semantic conformance.** We do not enforce RFC 3261 §8.1 mandatory headers (From, To, Call-ID, CSeq, Via, Max-Forwards). The Tier-1 parser does not require them; if a higher-level oracle eventually does, that's a separate property.
- **No invalid-input adversarial generation.** We're not trying to make first-parse fail. Adversarial bytes are the libfuzzer target's job. proptest's leverage is on the *valid* corner cases.
- **No Tier-2 typed-header round-trip.** `From::parse(value).map(|t| t.to_string()) == value` is **Track B's** invariant. Track A only asserts `Message::parse ∘ Message::to_bytes` Tier-1.
- **No `\n`-only line-ending generation.** Canonical CRLF on the wire only. The `\n` tolerance path is exercised by static fixtures.
- **No header-fold generation.** RFC 3261 §7.3.1 line-folded headers. The fold-collapse normalization is exercised by `tests/fixtures/handcrafted/invite_folded_subject.sip` and the static fixture's oracle invocation. proptest generating folds adds combinatorics for marginal Tier-1 signal.
- **No multi-Via stacking variability.** Real responses have N Via headers; we let `arb_header_set` produce 0–12 of any header (including multiple Vias by chance), but we do not specifically force the multi-Via path. The static fixture `rt_handcrafted_response_with_multi_via` already pins it.

## 8. Open questions for the supervisor

**Q1 (load-bearing).** Should `arb_method_token`'s case-variant tail (10% of `iNvItE`-style mixed-case methods) be retained? It tests that `Method::FromStr` is case-insensitive AND that the serializer canonicalizes case (`Method::as_str` always returns uppercase, so m1 ≠ m2 in bytes). The fixed-point oracle handles this — m2 == m3 — but it's a different code path than the canonical-input case. **My recommendation: keep it (cheap, real behavior, covered by `framing.rs:test_request_line_*` unit tests but proptest exercises the full Message round-trip).** Supervisor decision needed.

**Q2.** The reason-phrase generator currently restricts to printable-ASCII to avoid the post-M11 parser tightening that rejects CTL bytes. Should we add a tail of high-bit UTF-8 (`UTF8-NONASCII / UTF8-CONT`) to exercise the parser's UTF-8 handling? The parser explicitly accepts these (`framing.rs:test_parse_status_line_reason_with_high_bit_accepts`). My instinct is yes — it's a 1-line strategy addition (`"[\u{80}-\u{10FFFF}]{0,8}".prop_map(...)`) and it surfaces any encoding mismatch in `to_bytes`. But this is supervisor's call.

**Q3 (deferred to a Track D, or absorb here?).** The current generator does not produce headers with leading whitespace before the colon (`"Foo : value"`), which the parser tolerantly accepts (`header.rs:test_parse_line_whitespace_before_colon`). The serializer always emits no-space-before-colon, so a `Foo :` input round-trips to `Foo:` — m1 differs from m2 in bytes, but m2 is a fixed point. This is a legitimate normalization the oracle handles. Worth adding a 10%-weight `Foo : value` channel? My recommendation: **no, defer**. It's covered by the static `rt_oracle_holds_after_normalization` test and by libfuzzer's mutation. proptest's leverage is elsewhere.

**Q4.** Do we ever want this proptest test to run against a *nightly*-built `cargo test` with `--release`? Currently it'll run debug, which is the default `cargo test` mode and matches the workspace-test bar. 256 cases in debug at ~0.1ms each is fine; if we crank `RSIPRTP_PROPTEST_CASES=1_000_000` for soak, debug at ~100s is acceptable but `--release` would be ~5s. Probably belongs in a `Makefile` or `xtask` recipe alongside the existing fuzz launcher. Supervisor: do we add a recipe now, or wait until Track B and C land and we build a `proptest-soak.sh` once?
