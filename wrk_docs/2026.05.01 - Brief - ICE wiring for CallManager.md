# 2026.05.01 - Brief - ICE wiring for CallManager

## One-line goal

Make ICE end-to-end usable from `CallManager` so a call placed from behind
a NAT can negotiate connectivity to its peer without the application
reaching into the `ice::` module manually.

## Why now

The library code is already there — three substantial files totalling
~8.5k lines (`ice/agent.rs`, `ice/stun.rs`, `ice/turn.rs`) implementing
RFC 8445 (ICE), RFC 5389/8489 (STUN), and RFC 5766/8656 (TURN). What is
missing is the **glue from the call layer**: nothing inside `session/`
references `ice::`, and the SDP builder has no first-class ICE support.

That glue is the difference between "this stack works on a public IP" and
"this stack works in the real world." Until it lands, anyone deploying
behind a typical home/office NAT is stuck building the integration
themselves from primitives.

The README currently lists this honestly under [Scope] as out of band.
This brief is the plan to close it.

## Current state — confirmed by grep

**ICE library (`src/ice/`)**

- `IceAgent::new(config, role)` — constructs an agent. Public API on
  `agent.rs:32` exports `IceAgent, IceConfig, IceRole, IceState,
  CandidatePair, PairState`.
- `agent.set_remote_credentials(ufrag, pwd)` — `agent.rs:190`.
- `agent.gather_candidates()` — async, returns `Vec<Candidate>`,
  `agent.rs:227`. Uses `IceConfig::stun_servers` (defaults to
  `StunServer::GOOGLE`).
- `agent.add_remote_candidates(Vec<Candidate>)` — `agent.rs:421`.
- `agent.start_checks()` — async, drives connectivity checks,
  `agent.rs:490`.
- `agent.selected_pair()` — `agent.rs:219`.
- `agent.local_candidates()` — `agent.rs:214`.
- `agent.state()` — current `IceState`, `agent.rs:209`.
- `agent.close()` — `agent.rs:805`.

**SDP layer (`src/sdp/`)**

- No dedicated ICE attribute support. `ice-ufrag` only appears in
  `builder.rs` test scaffolding (lines 788, 797, 854) where it is added
  through the generic `Attribute { name, value }` API.
- No `Candidate <-> SDP` parser/serializer. `a=candidate:...` lines are
  preserved as raw attributes by `parser.rs` but not turned into typed
  values.

**Session layer (`src/session/`)**

- `manager.rs` has zero references to `ice::` — confirmed by grep on
  2026-05-01.
- `Call` (in `session/call.rs`) holds a `MediaSession` but has no
  candidate/agent fields.
- `MediaSession` allocates a single local RTP port via
  `CallManager::allocate_rtp_port()` and embeds it directly into SDP. No
  hook for "the local address might change after ICE completes."

## Out of scope for this brief

These are deliberately excluded — they need their own briefs once the
basic wiring lands:

- **Trickle ICE** (RFC 8838). The current agent gathers candidates in a
  batch; trickle requires interleaving gathering with offer/answer. Land
  full ICE first, then add trickle as an opt-in.
- **ICE restart** (re-INVITE with new ufrag/pwd). Real-world but rare.
- **TURN credential refresh** for long calls. The `TurnClient` does
  allocations; refresh-on-expiry is a follow-up.
- **DTLS-SRTP**. Separate brief — see
  `2026.05.01 - Brief - DTLS-SRTP decision.md`. ICE wiring should not
  block on DTLS; SDES + ICE is a coherent feature on its own.

## Desired end state

A user does this:

```rust
let mut config = ManagerConfig::default();
config.ice = Some(IceManagerConfig {
    stun_servers: vec![StunServer::GOOGLE],
    turn_servers: vec![],          // optional
    role_strategy: IceRoleStrategy::FromOfferAnswer,
});
let mut manager = CallManager::new(config);
let call_id = manager.create_call("sip:bob@example.com".to_string());

// Manager emits ManagerEvent::IceGatheringComplete(call_id, sdp_offer)
// when local candidates are ready, then drives checks against remote
// candidates parsed from the answer. ManagerEvent::IceConnected fires
// when a pair is nominated; the application now has a usable peer
// SocketAddr for RTP.
```

ICE is **opt-in via config** — calls without ICE configured continue to
work exactly as today.

## Plan

### Phase 1 — typed SDP attributes

Add typed parsing/building for the four headline ICE attributes so the
rest of the stack does not have to handle `Vec<Attribute>` strings.

Files: `src/sdp/parser.rs`, `src/sdp/builder.rs`.

- Add `IceCandidate` struct (foundation, component, transport, priority,
  address, port, type, related-addr/port, generation).
- Parse `a=candidate:` per RFC 8839 §5.1.
- Add `a=ice-ufrag`, `a=ice-pwd`, `a=ice-options` as typed fields on
  `MediaDescription` (and session-level fallback per RFC 8839 §4.1).
- Round-trip tests: parse → build → parse must be lossless.

**Acceptance:** `cargo test -p rsiprtp sdp::parser::tests::ice` and
`sdp::builder::tests::ice` pass; round-trip test against a captured Linphone
or PJSIP offer/answer.

### Phase 2 — `IceMediaCoordinator`

Introduce a thin per-call wrapper that owns the agent and the
RTP/RTCP `UdpSocket`s the agent needs for STUN. This is the integration
seam — `Call` holds one of these instead of holding the agent directly.

File: new `src/session/ice_media.rs`.

Responsibilities:

- Bind UDP sockets in the configured RTP port range.
- Construct `IceAgent` with `local_ufrag`/`local_pwd`.
- Drive `gather_candidates()` and surface results as a stream of
  `IceCandidate` values ready for SDP serialization.
- Accept remote candidates from a parsed SDP and pass them to the agent.
- Drive `start_checks()` and emit a `Connected(SocketAddr)` event when a
  pair is nominated, surfacing the peer address the RTP session should
  send to.
- Hold the sockets and hand them to the RTP session **after** ICE
  succeeds — the RTP session needs to share the same `UdpSocket` that
  ICE used (or at least the same local port) for symmetric RTP to work.

**Acceptance:** unit tests with a fake STUN server (we have `StunClient`
tests already) — coordinator gathers, exchanges candidates, drives
checks, reports `Connected`.

### Phase 3 — `CallManager` integration

Files: `src/session/manager.rs`, `src/session/call.rs`.

- Extend `ManagerConfig` with `ice: Option<IceManagerConfig>`.
- When ICE is configured, `create_call` and `handle_incoming_invite`
  construct an `IceMediaCoordinator` instead of allocating a bare RTP
  port.
- Add `ManagerEvent::IceGathering`, `ManagerEvent::IceConnected`,
  `ManagerEvent::IceFailed` variants. The application drives the
  coordinator the same way it drives the existing transaction loop.
- Role selection per RFC 8445 §6.1.1: offerer is controlling by default;
  on collision (both controlling/both controlled), tie-break on
  tie-breaker.
- SDP offer is delayed until candidate gathering completes (or trickled
  later — out of scope for Phase 1).

**Acceptance:** new integration test `tests/ice_basic.rs` placing a call
between two `CallManager` instances on `127.0.0.1` with ICE enabled —
both sides gather host candidates, exchange them via SDP, complete
checks, and start RTP.

### Phase 4 — examples and docs

- New example: `crates/rsiprtp/examples/ice_call.rs`.
- README: move ICE bullet from "out of scope" to Features (qualified:
  "host + server-reflexive; trickle and ICE restart are not yet
  supported").
- Update `lib.rs` top-level doc.

**Acceptance:** `cargo build --examples` clean; example runs end-to-end
in a test container with a STUN server reachable.

## Open decisions

1. **Async surface.** The current agent API is `async`, but
   `CallManager` is Sans-IO. The cleanest fit is to keep the coordinator
   `async` (it owns sockets) and have `CallManager` emit events that the
   application's event loop pumps to/from the coordinator. Confirm this
   shape before Phase 2.
2. **TURN by default?** STUN is cheap and almost always sufficient for
   home-NAT cases. TURN requires credentials and a server. Default config
   should probably be STUN-only with TURN purely opt-in.
3. **rtcp-mux.** ICE pairs are per-component (RFC 8445). With `rtcp-mux`
   we have one component (RTP+RTCP shared). Without, we have two. We
   should require `rtcp-mux` for ICE — most callers want it anyway, and
   it halves the gathering cost.
4. **Aggressive vs regular nomination.** RFC 8445 §8.1.1 default is
   regular. The agent supports both today. Keep regular as the default.

## Risk register

| Risk | Likelihood | Impact | Mitigation |
|------|------------|--------|------------|
| Socket-sharing between ICE and RTP is fiddly on Windows | Med | High — RTP fails after ICE succeeds | Prototype socket handoff in Phase 2; have `tokio::net::UdpSocket::from_std` path ready |
| SDP attribute round-trip tests reveal divergences vs major UAs | Med | Med — interop fix-up | Capture real offers from baresip, Linphone, Asterisk early |
| `IceConfig::default()` uses Google STUN — not appropriate for some deployments | High | Low | Default to empty `stun_servers` once ICE is opt-in; require explicit config |
| Long gather timeouts (default 5000ms) block call setup visibly | Med | Med | Make gather timeout part of `IceManagerConfig`; document the trade-off |
