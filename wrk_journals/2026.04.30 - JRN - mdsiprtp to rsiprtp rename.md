### Test isolation bug — left for follow-up

The `force_*_error` test pattern uses `static AtomicU64` flags storing thread
IDs to inject errors into specific code paths. With parallel test execution,
flags set in one test can be observed by other tests on the same thread (or
similar-hashing thread), causing both flake-failures of the test that set
the flag AND failures of unrelated tests that consume the leaked flag.
Affects roughly: `rsiprtp-transport::{tcp,tls}`, `rsiprtp-ice::turn`.
Should be tracked as a separate fix later.

### Surprise: `master` branch had reappeared on GitHub at `c98e12e` (pre-scrub)

During pre-push fetch, `git ls-remote origin` showed `refs/heads/master`
pointing at `c98e12e8c0b916d84da604c74eb0e260ed5e4ea6` — that's the
**pre-scrub HEAD** whose ancestry includes commits `372c16f` and `0bf7e65`,
both of which carried the test PEM keys we redacted on 2026-04-28.

Earlier in this project I had:
1. Force-pushed scrubbed history to `origin/main` (overwriting the dirty
   default branch).
2. Created a transient `master` branch at the post-scrub HEAD and then
   deleted it.

So `master` was gone after the scrub. Something pushed it back at the
**pre-scrub** SHA `c98e12e` between then and now. Most likely cause: a
clone elsewhere (an agent / a separate working tree) that still had the
unscrubbed local history ran `git push origin <pre-scrub-ref>:master`.
Hard to confirm without admin audit logs.

**Action taken:** `git push origin --delete master`. Verified
`git ls-remote origin` now shows only `refs/heads/main`. Pruned the local
stale `origin/master` ref.

**Caveat:** GitHub retains deleted-branch commits as ghost objects for
~90 days; anyone with the SHA can still fetch them. Forks (if any) keep
the ref. The keys were test fixtures (never authenticated anything) so
this is hygiene rather than crisis, but worth flagging to Arthur — and
worth understanding **how** master got pushed back, because if the cause
is still active it'll re-leak.
