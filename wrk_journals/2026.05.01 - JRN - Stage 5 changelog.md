## Judgement calls on wording

1. **Format glyph: em-dash, not hyphen, between version and date.** Keep a
   Changelog spec writes `## [1.0.0] - 2017-06-20` with an ASCII hyphen.
   I used `—` (em-dash) because that's what the user spec for this stage
   asked for ("`## [0.2.0] — 2026-05-DD`"). Either is valid; followed the
   instruction.

2. **MSRV bump bolded.** Devil's advocate flagged this as "the most likely
   thing a downstream user trips on." Bolded the headline phrase only,
   not the whole bullet — keeps scannability without screaming.

3. **DTLS honesty.** The user spec required the entry say "DTLS-SRTP
   framing types remain; the handshake itself is not yet implemented."
   Reworded slightly to flow naturally: "the DTLS-SRTP framing types
   remain in `rsiprtp::srtp`; the handshake itself is not yet implemented,
   so there was nothing for `openssl` to gate." The "nothing for openssl
   to gate" half-sentence justifies the removal in one beat.

4. **Empty `[Unreleased]` categories.** Spec says "empty list per category."
   Listed all six standard Keep a Changelog categories
   (Added/Changed/Deprecated/Removed/Fixed/Security) so a future PR
   author doesn't have to remember which categories exist. Headers only,
   no placeholder bullets — Markdown renderers handle empty H3s fine.

5. **Stage 4 idiom cleanups: one bullet under Changed.** Spec said "one
   line is plenty." Resisted the urge to split into three bullets.

6. **Header note above first section.** Two short lines: format + semver
   commitment. The semver line is a freebie — costs one line, signals
   intent for downstream consumers, and `0x4D44/rsiprtp` is genuinely
   semver-versioned.

## What I deliberately did not add

- **MSRV-stability policy in README.** Devil's advocate suggested
  considering this. Out of scope for Stage 5; the CHANGELOG entry alone
  satisfies the "must call out the bump" requirement. README policy is a
  separate decision (and a separate commit).
- **Unreleased compare link target tweaks.** The link points at
  `v0.2.0...HEAD`. Once the v0.2.0 tag is cut, GitHub renders the diff
  correctly. Until then the link 404s — acceptable; same trade-off applies
  to the `[0.2.0]` release-tag link, and the spec accepted that.
- **Older versions section.** This is the first changelog entry. There is
  no 0.1.x history to backfill (0.1.0 was an internal pre-publish baseline
  per the publishing plan in MEMORY.md). Starting at 0.2.0 is honest.
