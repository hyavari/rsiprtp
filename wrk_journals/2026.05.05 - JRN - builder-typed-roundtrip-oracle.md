## Plan

Implement the typed → serialize → parse → compare oracle described
in the HLD. One new file (`crates/rsiprtp/tests/builder_roundtrip.rs`),
17 fixtures, ~220–250 LOC, no changes to existing code.

Stage gating:

- **S1 — Implement.** Agent writes the test file with helpers + 17
  fixtures per HLD §"Fixture set". Runs the suite. Reports
  pass/fail per fixture. **Halt before triaging any failure** —
  supervisor decides if it's a fixture-canonicalization issue (fix
  fixture), a builder bug (decision: fix now or pin?), or a typed
  accessor bug (same decision).
- **S2 — Lint.** clippy + fmt on the new file only.
- **S3 — Review.** code-reviewer agent + devil's-advocate agent in
  parallel. Both read the HLD first, then the implementation. Look
  for: missed accessor coverage, fixtures that pass for the wrong
  reason (Goodhart), brittle assertions, precondition violations.
- **S4 — Apply feedback.** code-review-apply-feedback agent
  consolidates legitimate findings.
- **S5 — Verify.** Final agent compares implementation against
  HLD fixture list; gating on 100% completeness.
- **Commits:** after S1 (working impl), after S4 (post-feedback),
  after S5 (verified).

### Lessons

- The devil's-advocate review was the highest-leverage stage. It
  found 6 high-signal items the code-reviewer missed, including
  three fixtures that were testing parser stability rather than
  builder emission. Worth keeping in the playbook for any future
  oracle work.
- "Every builder setter with a typed accessor gets a fixture" is
  the rule that turned what could have been a Goodhart pile into
  honest coverage. The HLD's explicit list of expected
  normalizations also prevented multiple wasted triage cycles.
- One thing the existing oracle code-review skill could be
  sharper on: detecting `r.to_bytes()` (post-round-trip)
  assertions when the intent is "test the builder emission". The
  charitable code-reviewer let three of these through.
