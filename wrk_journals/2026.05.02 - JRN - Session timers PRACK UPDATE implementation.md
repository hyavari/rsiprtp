## Log

### Phase 1 — SIP headers — DONE
- Implementer: 6 header types + Refresher enum in `headers.rs`; accessors and
  builder methods on Request/Response in `message.rs`; re-exports in `mod.rs`.
  39 + 30 unit tests added (round-trip, malformed, builder semantics).
- Reviewer (code-reviewer methodology): flagged multi-line `Require:`/`Supported:`
  not coalesced (RFC 3261 §7.3.1) — load-bearing for Phase 2 PRACK detection.
  Other findings advisory.
- Apply-feedback round 1: fixed multi-line merge in `find_require`/`find_supported`,
  added two coalescing tests. +2 tests, 424 passing.
- Devil's advocate: surfaced 3 real interop bugs:
  1. Doc lie on `require()`/`supported()` setters claiming Display lowercases
     (it doesn't).
  2. `SessionExpires::parse` rejects whitespace around `=` (RFC 3261 EQUAL
     allows SWS) → silently drops refresher choice → Phase 4 trip wire.
  3. Compact form `x:` for `Session-Expires` (RFC 4028 §4) not recognized.
  Plus one low-impact note on `RAck::method` enum lock-in (skipped per scope).
- Apply-feedback round 2: fixed all three. +3 tests, 427 passing.
  Build + clippy clean.
- Status: ready to commit.

### Phase 2 — PRACK transaction — DONE
- Implementer: `send_provisional_reliable`, `Timer::N`, `Event::PrackReceived`,
  `Event::PrackTimeout { rseq, final_sent: FinalSent }`, `FinalSent` enum, buffer
  drain on any final response. Started with byte-level header injection
  because the brief restricted `sip/message.rs`. 14 new tests.
- Reviewer: flagged two real bugs:
  1. Byte injection duplicates `Require:` if TU pre-set one.
  2. Timer N model conflated `next_interval` with per-entry remaining time;
     surviving entries fired prematurely on next tick.
  Plus quality items: round-trip-parse test missing, FinalSent variants undocumented,
  `next_rseq.saturating_add` should be `checked_add`.
- Apply-feedback round 1:
  1. Added `pub fn SipResponse::inner_mut()`; rewrote `stamp_reliable_headers` to
     mutate typed `rsip::Headers`; merges existing `Require:` (case-insensitive
     dedupe of `100rel`) per RFC 3261 §7.3.1; debug-asserts no pre-existing RSeq.
  2. `ReliableProvisional` gained `next_fire_after` and `abandon_at`; per-entry
     scheduling correct.
  3. Round-trip-parse test, FinalSent doc, RSeq overflow check.
  +3 tests, 2687 passing.
- Devil's advocate: 6 problems. Real:
  1. `send_provisional_reliable` unconditionally emits `SetTimer(N, T1)` even when
     a Timer N is already pending — schedule drift with staggered creation.
  2. The new test stages both entries simultaneously; doesn't actually
     exercise independent schedules (test theatre).
  3. `pub fn inner_mut()` too permissive — should be `pub(crate)`.
  Phase-4 follow-ups (acceptable as-is until Phase 4 lands):
  - `PrackTimeout` event silently dropped in manager stub (Phase 4 wires it).
  - Dead `PrackReceived` half in manager match arm (Phase 4 removes it).
- Apply-feedback round 2:
  1. `send_provisional_reliable` only emits `SetTimer(N, T1)` when buffer was
     empty. Comment notes RFC-conformant trade-off (T1 is a minimum).
  2. Test renamed to `test_multiple_outstanding_prack_staggered`; stages entry1,
     drives Timer N once, stages entry2 mid-flight, drives to abandonment;
     asserts head-start invariant + buffer-drain.
  3. `inner_mut` -> `pub(crate)`.
- Status: 2687 passing, build + clippy clean. Ready to commit.

### Phase 3 — Dialog methods — DONE
- Implementer: `build_prack`, `build_update`, `handle_update` per HLD deliverable 4.
  9 new tests, 170 dialog tests pass. Two flagged deviations:
  1. Allow on outbound UPDATE punted to Phase 4 (no `allow()` setter on
     `SipRequestBuilder`).
  2. Pre-existing route-set bug — `build_in_dialog_request` ignored route_set;
     PRACK/UPDATE/BYE through any record-routing proxy would have been
     misrouted.
- Reviewer (code-reviewer methodology): caught both, noted Allow on UPDATE
  is a 5-line addition (not Phase-4 territory), noted route-set is a real
  interop hole that Phase 4 will rely on; tests for `test_send_bye_with_route_set`
  weren't asserting the actual Route header.
- Apply-feedback round 1:
  1. Added `allow()` setters + accessors to both Request and Response builders
     and accessor sides. `build_update` and `handle_update` now use them
     symmetrically; `inner_mut()` injection in handle_update dropped.
  2. Loose-route implementation in `build_in_dialog_request` (RFC 3261
     §12.2.1.1 modern path); `route()` builder setter + `route_headers()`
     accessor added. BYE picks this up automatically via the shared helper.
  3. `cseq_method().unwrap_or` → `expect`.
  +7 tests; 1 existing test fixed; 2702 passing.
- Devil's advocate: 3 problems, all pre-existing in the route-set parser:
  1. `RecordRoute::lr` only detects `;lr` *after* the `<>`, not inside.
     `<sip:proxy;lr>` (canonical loose form) reads back as `is_loose_routing()
     == false`. Latent — Phase 4 doesn't gate on this flag.
  2. Docstring on `build_in_dialog_request` claimed strict-route input "will
     fail visibly in tests" — no such mechanism exists.
  3. `params.contains("lr")` would false-positive on `lro` / `color=lrx`.
     Latent.
- Fix applied: docstring updated to be honest about strict-route silent emission
  (#2). #1 and #3 are pre-existing latent bugs not used by Phase 4 — captured
  here as known limitations, not addressed in this work.
- Status: 2702 passing, build + clippy clean. Ready to commit.

### Known latent bugs (deferred — not blocking Phase 4)
- `RouteSet::is_loose_routing()` returns false for canonical `<sip:proxy;lr>`
  (lr-inside-URI). Fix is a `RecordRoute` parsing enhancement.
- `RecordRoute` `params.contains("lr")` false-positives on `lro`, `color=lrx`.
- `build_in_dialog_request` silently emits loose-route-shaped requests against
  strict-route peers (no detection, no error). Acceptable for 2026 reality.
