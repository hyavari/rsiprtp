# Stage 4 — MSRV `--locked` build

## Goal

HLD item 18: ensure `cargo build --workspace --exclude gabby --locked`
succeeds under the workspace MSRV. Decide between pinning a 1.75-compatible
version of the offending dep (Path A) or formally bumping MSRV (Path B).

## Diagnosis

Rust 1.75 was not installed locally. Installed it: `rustup toolchain
install 1.75` (rustc 1.75.0, 2023-12-21).

Initial `cargo +1.75 build --workspace --exclude gabby --locked` failed
immediately at lockfile parse:

    error: failed to parse lock file at: D:\language\rsiprtp\Cargo.lock
    Caused by:
      lock file version `4` was found, but this version of Cargo does not
      understand this lock file, perhaps Cargo needs to be updated?

Lockfile v4 was introduced in Cargo 1.78. So the existing `Cargo.lock`
already exceeds 1.75. To see what the *deps* require, I regenerated the
lockfile under 1.75 (`cargo +1.75 generate-lockfile`) — that picks the
newest semver-compatible versions of every dep — then built. That failed
deeper:

    error: failed to parse manifest at `...\time-0.3.47\Cargo.toml`
    Caused by:
      feature `edition2024` is required ...
      not stabilized in this version of Cargo (1.75.0)

`edition2024` was stabilized in Cargo 1.85. Restored the original lockfile
and tried 1.85 directly:

    error: rustc 1.85.1 is not supported by the following packages:
      ropus@0.12.0 requires rustc 1.88
      time@0.3.47 requires rustc 1.88.0
      time-core@0.1.8 requires rustc 1.88.0

So the binding constraints are:
- `ropus 0.12.0` — direct workspace dep (recently bumped, commit
  `f5e0c44` "Add SessionCodec sum type with AdaptiveBitrate dispatch").
  Crate states `rust-version = "1.88"`.
- `time 0.3.47` — transitive via `ezk-g722`, `rcgen`, `yasna`. Crate
  states `rust-version = "1.88.0"`.

Tested 1.88 directly: `cargo +1.88 build --workspace --exclude gabby
--locked` succeeds cleanly.

## Decision: Path B (formal MSRV bump to 1.88)

**Reasoning.** Path A (pin) was not viable here:

1. **Lockfile v4** would have to be regenerated to v3, which would force
   downgrading every recently-resolved dep — potentially many of them —
   to versions whose Cargo metadata predates v4.
2. **`ropus` is a direct dep** we explicitly upgraded to 0.12 in the
   recent SessionCodec work. Pinning to an earlier major would mean
   undoing that work or accepting a different codec API.
3. **`time` is pulled by three separate transitive deps** (`ezk-g722`,
   `rcgen`, `yasna`), each potentially with different compatible-version
   ranges. Pinning `time` would likely cascade.
4. **No security concern with Rust 1.88.** Released 2025-06-23 — about
   10 months stable as of today (2026-04-30), well past the 6-month
   threshold mentioned in the stage instructions for "default to the
   lockfile-required MSRV rather than latest stable".

The HLD's stage-4 instructions say: "Default to bumping to the lowest
rustc that satisfies the lockfile — don't preemptively go to 'latest
stable.'" 1.88 is the lowest that satisfies the lockfile (`ropus`'s
rust-version field is hard-coded to 1.88). Cargo's error messages were
explicit: 1.85 fails, 1.86/1.87 would also fail by the same error, 1.88
is the floor.

## Files changed

1. **`Cargo.toml`** — `[workspace.package] rust-version = "1.75"` →
   `"1.88"`.
2. **`README.md`** — MSRV badge `rustc-1.75+` → `rustc-1.88+`; "MSRV:
   **Rust 1.75**" → "MSRV: **Rust 1.88**".
3. **`.github/workflows/ci.yml`** — job name `MSRV (Rust 1.75)` →
   `MSRV (Rust 1.88)`; `dtolnay/rust-toolchain@1.75.0` →
   `@1.88.0`.
4. **`crates/rsiprtp/src/rtp/rtcp.rs`** — restored idiomatic
   `is_multiple_of` (two sites). Stage 2 had reverted to `% 4 != 0`
   manual form because `is_multiple_of` was stabilized in 1.87. Now MSRV
   is 1.88, so the idiomatic version is back. Without this, `cargo
   clippy ... -D warnings` fails on toolchain 1.94 (the project's
   default-stable) with `clippy::manual_is_multiple_of`.
5. **`crates/rsiprtp/src/rtp/session.rs`** — restored
   `Duration::abs_diff` (one site). Same reasoning — Stage 2 had reverted
   to a manual `if a > b { a - b } else { b - a }`. `Duration::abs_diff`
   was stabilized in 1.81; safely available under MSRV 1.88. Without
   this, clippy fails on `clippy::manual_abs_diff`.
6. **`crates/rsiprtp/tests/baresip_integration/framework/config.rs`** —
   restored `port.is_multiple_of(2)` (one site). Same Stage 2 revert,
   same reason to restore.

I did not touch `wrk_journals/2026.05.01 - JRN - Single-crate collapse
implementation.md` — it was modified by another agent (the supervisor)
in-flight when I started, and per multi-agent policy I left it alone.

## Notes / leftover risks

- The four reverted clippy fixes (`is_multiple_of` ×3, `abs_diff` ×1) are
  technically out of Stage 4's stated scope, but they were direct
  consequences of the MSRV bump: leaving them as the manual forms would
  have made `cargo clippy ... -D warnings` fail on the project's stable
  toolchain (1.94). The whole point of Stage 4 is `--locked` build
  health; leaving clippy broken would have failed the validation matrix.
- The original `Cargo.lock` (v4) was committed before this stage — Stage
  1's MSRV pin to 1.75 was source-only; nobody ran `--locked` build
  against 1.75 before now. Stage 4 surfaces and resolves that gap.
- The `unused manifest key: workspace.lints.rust.unexpected_cfgs.check-cfg`
  warning under 1.75 (no longer relevant, since we abandoned 1.75) is a
  separate Cargo-version concern that vanished with 1.88.
- I did not bump `crates/rsiprtp/Cargo.toml` because it inherits via
  `rust-version.workspace = true`.
- Gabby is intentionally untouched (it's `--exclude gabby` in CI MSRV
  job and requires Vosk anyway).
