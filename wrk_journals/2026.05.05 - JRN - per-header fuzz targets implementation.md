## Plan summary

Add four crash/UB libfuzzer targets in `fuzz/fuzz_targets/`:
- `sip_via_typed` → `parser::typed::Via::parse`
- `sip_contact` → `parser::typed::Contact::parse`
- `sip_name_addr` → `parser::name_addr::NameAddr::parse`
- `sip_cseq` → `parser::typed::CSeq::parse`

Plus four `[[bin]]` stanzas in `fuzz/Cargo.toml`. ~30 LOC across 4 new
files + ~28 lines of Cargo.toml. No changes to existing code.

## TDD note

"test" is `cargo +nightly fuzz build` succeeding plus a brief smoke
run. Calling this out so we don't pretend TDD applies in a contrived
way. The HLD itself is the upfront design artefact; the existing
`from.rs` / `to.rs` / `cseq.rs` test suites already cover semantic
behavior of the parsers.

**Devil's-advocate:** code clean; pushed hard on two narrative gaps.

1. *"Nobody runs these targets."* `fuzz_overnight.ps1` only runs

   `sip_message_parse`; CI runs no fuzz at all. The HLD explicitly

   wire one in now or be honest in the journal/commit that this is
   build-only scaffolding pending campaign planning.
2. *`sip_name_addr` mostly redundant with `sip_contact`.*
   Contact's non-wildcard branch is a literal `NameAddr::parse(trimmed)`
   call. >95% mutation overlap. HLD's "decide after first 8h soak"
   open question is unscheduled. Pick one: drop `sip_name_addr`
   now, or commit to the soak.

**Non-blocking observations** (worth recording, not changing code):

3. `cov: 148 ft: 353` on `sip_cseq` after 6.7M execs is "this
   grammar has two tokens and an integer" — confirms wired up, says
   nothing about bug-finding power. No retirement criterion in the
   HLD; future campaign owner should write one before scheduling.
4. `from_utf8` filter is *justified*, not vestigial: `framing.rs:39`
   already requires the header section to be valid UTF-8
   (`SipError::Parse` on `from_utf8` failure), so production-reachable
   bytes for these parsers are always UTF-8. The filter discards no
   bug-finding pressure.
5. `#[allow(dead_code, unused_imports)]` on `parser/mod.rs:18` mutes
   variant-level dead code, *not* the parsers themselves. The fuzz
   targets reach the same `parse` functions that `SipMessage::parse`
   does in production via `framing.rs` / `header.rs` `.typed()`
   paths. Not fuzzing dead code.

### Adjudication

Zero code changes warranted. Both stop-the-line concerns are
narrative honesty issues, not implementation defects:

- **#1 (no runner):** the HLD explicitly puts supervisor wiring out
  of scope. Expanding scope here is the wrong call — but the
  devil's-advocate is right that the journal and commit message
  should not pretend a 60s smoke run is meaningful coverage.
  Adjusting framing accordingly.
- **#2 (`sip_name_addr` redundancy):** defending the design. The
  per-binary coverage map is what makes a focused target valuable,
  not the underlying parser code. `sip_contact` spends mutation
  cycles gating on the wildcard `*` discriminator, libfuzzer
  prioritizes inputs that hit new edges in *that binary*, and corpus
  separation matters for triage. The HLD's open-question hook stays
  in place; if the soak shows zero divergent value, fold then. The
  cost of being wrong is one binary stanza.

Stage 2 outcome: adjust narrative, fix HLD nit (`cargo fuzz build`
doesn't accept multi-target args), no code edits.
