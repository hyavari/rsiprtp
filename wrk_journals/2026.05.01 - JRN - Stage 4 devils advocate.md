# Stage 4 — MSRV bump devil's advocate review

Reviewer: adversarial agent. Implementer just bumped workspace MSRV from
1.75 to 1.88. Question: is this defensible?

## TL;DR

**1.88 is genuinely the floor.** Confirmed empirically. The only
alternative path (downgrading `ropus` to 0.5.0, the last version that
supports 1.85) would require ripping out the typed `Bitrate` /
`InbandFec` / `set_bitrate` / `set_fec` / `set_dtx` API the codebase
actively depends on (commits f8293a6 + f5e0c44 + the BitrateBridge
work).

The implementer's source-file reverts (re-introducing `is_multiple_of`
and `Duration::abs_diff`) are correct and necessary — without them,
`cargo clippy -D warnings` on the project's stable toolchain (1.94)
fails.

The bump is **internally consistent** across `Cargo.toml`, `README.md`
(both badge URL and body text), and `.github/workflows/ci.yml` (both
job name and toolchain version). All historical 1.75 references in
journals/HLDs are correctly left as historical record.

**Open issue: README has no MSRV-stability policy and no CHANGELOG
entry yet.** That's deferred to Stage 5 (CHANGELOG creation), but if
the project ever publishes to crates.io, jumping a published MSRV from
1.75 to 1.88 in a 0.2.x release without a policy note would be a
"surprise upgrade" for any downstream consumer.

## 1. Is 1.88 truly the floor?

### Sub-claim a — could we pin `ropus = "0.11"` instead?

**Tested. No.** The implementer's note that 0.11 would help is wrong.
crates.io API confirms:

- `ropus 0.12.0` — rust-version `1.88`
- `ropus 0.11.1` — rust-version `1.88`
- `ropus 0.5.0` — rust-version `1.85`

The 0.6 → 0.11 series all require 1.88. Only `ropus 0.5.0` would let us
hit 1.85.

But **0.5.0 lacks the typed runtime API**. Commit f8293a6 ("Bump ropus
to 0.12") added `set_bitrate`, `set_fec`, `set_dtx`, `set_packet_loss`
and replaced `OpusConfig.bitrate: u32` with `Bitrate` (an enum with
`Auto`/`Max`/`Bits(N)`). These are used by:

```
crates/rsiprtp/src/session/session_codec.rs
crates/rsiprtp/src/session/bitrate_bridge.rs
crates/rsiprtp/src/session/call.rs
crates/rsiprtp/src/media/codec/opus.rs
```

…which are all live, called paths (the BitrateBridge maps REMB events
into runtime `set_bitrate` calls on the Opus encoder). Reverting to
0.5.0 means undoing this whole feature set. **Path A (pin 0.11) is not
viable** — but for a different reason than the implementer stated.

### Sub-claim b — could we pin `time` lower?

**Tested. No.** `time 0.3.46` also requires 1.88. `time 0.3.41` only
requires 1.67.1, but `cargo +1.85 build --locked` with `time` pinned to
0.3.41 still fails with:

    error: rustc 1.85.1 is not supported by the following packages:
      ropus@0.12.0 requires rustc 1.88

So even if we could pin `time` (which would cascade through `deranged`,
`num-conv`, `time-core` — all downgraded), we'd still hit the `ropus`
floor. `time` is not the binding constraint.

### Sub-claim c — regenerate Cargo.lock at v3?

Pointless. Lockfile version is a Cargo-version concern, not a
rustc-MSRV concern. It would only let us *try* with older Cargo, which
would then fail on `time`'s `edition2024` flag. And we'd still hit
`ropus`. The lockfile v4 is a cosmetic side effect — the real binding
constraint is `ropus`'s `rust-version` field.

### Verdict

**1.88 is the lowest defensible MSRV** given current direct-dep
choices. The HLD's "default to lowest rustc the lockfile requires"
guidance is satisfied.

## 3. Are the source-level clippy reverts safe?

**Yes.** The three reverts are:

1. `crates/rsiprtp/src/rtp/rtcp.rs` — two sites of `% 4 != 0` →
   `!buf.len().is_multiple_of(4)`. `is_multiple_of` was stabilized in
   1.87. Under MSRV 1.88, available. Semantics: identical for any
   `usize`.
2. `crates/rsiprtp/src/rtp/session.rs` — manual abs-diff if/else →
   `self.smoothed_rtt.abs_diff(rtt)`. `Duration::abs_diff` was
   stabilized in 1.81. Under MSRV 1.88, available. Semantics:
   identical for `Duration`.
3. `crates/rsiprtp/tests/baresip_integration/framework/config.rs` —
   `port % 2 == 0` → `port.is_multiple_of(2)`. Same lint as #1.
   Available under 1.88.

All three are **idiomatic improvements** that the project's stable
clippy (1.94) lints for. Without them, `cargo clippy -D warnings`
fails on the project's CI clippy job (which uses stable, not 1.88).

Validation:

| Check | Result |
|---|---|
| `cargo +stable clippy --workspace --exclude gabby --all-targets -- -D warnings` | clean |
| `cargo +1.88 build --workspace --exclude gabby --locked` | clean |

Note: `cargo +1.88 clippy ... -D warnings` produces 208 errors. **This
is not a problem** — those are all `uninlined_format_args`, a clippy
1.88-era lint that fires across the codebase. The CI MSRV job runs
`cargo build`, not clippy, so this is irrelevant to gating. The
project's clippy gate is on stable (1.94+), not on MSRV. Standard
practice; no action needed.

## 5. Public-facing impact / README status note

**Gap.** The README's status section says:

> `rsiprtp` is **pre-1.0**. The public API will change between minor
> releases until 1.0. It is suitable for prototyping and serious
> internal use today, but pin an exact version before depending on it
> from production code.

There's **no policy statement on MSRV stability**. Common patterns:

- "MSRV bumps may happen in any 0.x release."
- "MSRV bumps are minor-version events."

Neither is asserted. For an unpublished project this is fine, but if
Stage 5 publishes 0.2.0 to crates.io, downstream consumers cloning
0.1.x → 0.2.x will discover the 1.75 → 1.88 jump only on build
failure. **Recommendation: Stage 5's CHANGELOG.md must call this out
explicitly under "Breaking" or "Notable" for 0.2.0**, e.g.:

> - **MSRV raised from 1.75 to 1.88.** Required by upgrades to
>   `ropus 0.12` (typed runtime bitrate API) and the `time 0.3.47`
>   transitive (via `ezk-rtp`).

The README's status text doesn't urgently need updating, but the
CHANGELOG entry is non-negotiable for a clean publish.

Single OS (Ubuntu). The `build-test` job runs on `ubuntu-latest`,

That's standard practice (MSRV gating on Linux only is universal in
the Rust ecosystem). No issue.

## 7. Is `time 0.3.47` runtime or dev-only?

**Runtime.** `cargo tree -e normal -p rsiprtp -i time` shows:

```
time v0.3.47
└── ezk-rtp v0.2.0
    └── ezk-g722 v0.1.1
        └── rsiprtp v0.2.0
```

`ezk-g722` is a **normal** dependency of `rsiprtp` (used for the G.722
codec). So `time 0.3.47` is part of the consumed library's dependency
graph, not just dev. The MSRV claim of 1.88 in the README accurately
reflects what downstream library users will need.

(`rcgen` and `yasna` also pull `time`, but those are dev-only — the
TLS test fixtures. They don't affect downstream library users.)

This means there's **no clever escape hatch** where we could claim a
lower "consumer MSRV" than "test MSRV". Both are 1.88.

## What might break this analysis later?

- If `ezk-rtp` ever drops `time` as a direct dep (or if upstream
  publishes a `time-pinned` 0.x line), MSRV could potentially drop.
  Worth re-checking on each `cargo update`.
- `ropus` is the project owner's own crate (`0x4D44/ropus`). The 1.88
  rust-version is a deliberate choice, not an upstream constraint. If
  the owner relaxed it, MSRV could drop without code changes.
