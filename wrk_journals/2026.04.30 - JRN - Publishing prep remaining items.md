## Open question for Arthur

**Publishing strategy needs revisiting.** The saved plan was
"single-facade publishing" — only `rsiprtp` goes to crates.io, the 11
internal crates stay `publish = false`. Mechanically this fails:

```
$ cargo publish -p rsiprtp --dry-run
error: all dependencies must have a version requirement specified when
publishing.
  dependency `rsiprtp-core` does not specify a version
```

Cargo refuses to publish a crate whose path-deps lack versions. To make
single-facade publishing work, we'd have to either:

- **A. Publish all 11 sub-crates** (drop `publish = false`, add `version`
  fields to workspace deps). Standard approach used by tokio, hyper,
  rustls, rsip itself. Each release ships 11 crates atomically.
- **B. Collapse internal crates into the facade** as modules. Loses the
  compile-time architectural enforcement Arthur wanted to preserve.
- **C. Vendor internal source into the facade at publish-time** via a
  build script. Ugly, fragile.

I'll continue Phase 1 metadata work (still correct regardless) and surface
this to Arthur before final validation. **Not a blocker for items
4/5/6/7/8/9/10.**

## Item 10 findings — `master` branch reappearance

Investigated as far as forensics allow without admin access to GitHub.

**On the local repo (`D:/language/rsiprtp/.git`):**
- `git fsck --lost-found` initially showed two dangling commits, including
  `c98e12e8c0b916d84da604c74eb0e260ed5e4ea6` — exactly the SHA `master`
  reappeared at on GitHub.
- This means the pre-scrub commit was still in this repo's object DB despite
  the 2026-04-28 GC. The earlier GC didn't aggressively prune unreferenced
  objects.
- Git config is clean: no hooks, no `remote.origin.push`, no aliases that
  would push `master`. Only `origin/main` is configured to fetch/push.

**Cleanup applied:**
- `git reflog expire --expire=now --all`
- `git gc --prune=now --aggressive`
- Post-cleanup `git fsck` shows no dangling objects.

**Filesystem search:** Only one clone of rsiprtp exists locally (under
`D:/language/`). No sibling clone found that could've held unscrubbed
history.

**Hypothesis (unprovable from this side):**
- Most likely cause: an old terminal session or background tool ran
  `git push origin master` against a local `master` ref that pointed at
  the pre-scrub HEAD. The push would've succeeded if `master` didn't yet
  exist on origin (GitHub's default branch protection wasn't on).
- Less likely: a separate machine / dev environment / CI runner had a
  cache and pushed master back.
- The local copy of `c98e12e` is now gone, so even if some old shell
  history references it, push won't find the object.

**Recommendation for Arthur:**
- **Enable GitHub branch protection on `main`:** require pull-request
  review, no force-pushes, no deletion. Settings → Branches → Add rule.
- **Enable secret scanning push protection** (Settings → Code security)
  so future leaks can't even reach the remote.
- **Consider a pre-receive hook** if you want to outright forbid `master`
  on this repo, but branch protection on `main` is usually sufficient
  since `master` is no longer the default.
- I can't prevent recurrence from outside; once branch protection is on,
  any rogue push attempt is logged and rejected.

This is documented but not closed-loop-fixed — flagging as a follow-up
for Arthur to act on via GitHub admin UI.

### Phase 4 — orchestrator-driven

- **Item 9 — test isolation fix (partial):** Agent `a6ef...` did the
  mechanical work — per-file `forced_error_lock()` helper added to 5 files
  (tcp/tls/udp/turn/stun); ~40 forced-error tests now acquire the lock as
  their first statement; `#[allow(clippy::await_holding_lock)]` added to
  test modules to silence lint.

  **The lock-only design doesn't fully solve the bug.** Bystander tests
  (e.g. `test_bind_server_success` which calls `bind_server` without
  taking the lock) can still consume forced flags set by another test.
  Five parallel runs reproduced deterministic races: `test_bind_server_*`,
  `test_udp_bidirectional` vs forced-error counterparts, etc.

  **Why ICE works in parallel but transport doesn't:** ICE production code
  checks `flag == current_thread_id()` before consuming, so flags set by
  one test don't fire errors in production code running on different
  threads. Transport just checks `flag != 0`, so any thread reading the
  flag triggers the error.

  **Action taken:**
  - Kept the lock additions — they prevent forced-error tests from racing
    each other within a file, which is a real (if partial) improvement.
  - Reverted the CI change — `--test-threads=1` is back. All 2780 tests
    pass serially.
  - Marked Task 15 completed-with-deferred-work. Proper fix is to
    convert tcp/tls/udp `FORCE_*_ERROR` statics from
    `AtomicU8`/`AtomicBool` to thread-id-keyed `AtomicU64`, mirroring
    what ice does. Tracked as deferred follow-up.
